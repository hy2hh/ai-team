# AI 에이전트 팀 직군별 도구(MCP) 매핑 설계 제안서

> 작성일: 2026-03-25
> 작성: Meeting Facilitator (전문가 패널 회의 결과)
> 상태: 제안 (sid 승인 대기)

---

## 1. 회의 개요

### 참석 전문가
- **Researcher**: 사용 가능한 MCP 도구 실태 조사
- **Architect**: 시스템 구조 및 도구 매핑 설계
- **Security Expert**: 권한 분리, 최소 권한 원칙
- **Product Manager**: 우선순위 및 ROI

### 성공 기준
1. 각 에이전트가 업무에 필요한 도구에 접근 가능
2. 최소 권한 원칙(Principle of Least Privilege) 준수
3. 현재 사용 가능한 MCP 서버로 즉시 실현 가능
4. 향후 도구 추가 시 확장 가능한 구조

## 3. 구현 시나리오

### Scenario 1: Conservative (Low Risk) - 최소 변경

**전략**: 현재 도구만 사용, 에이전트 persona 파일의 `tools:` 필드만 수정

**구현**:
- Atlassian MCP 읽기 도구를 모든 에이전트에 추가
- Atlassian MCP 쓰기 도구는 PM에게만 추가
- GitHub 접근은 기존 Bash 보유 에이전트(Frontend, Backend, SecOps)만
- Notion/Figma는 보류

**장점**:
- 즉시 적용 가능 (persona 파일만 수정)
- 기존 아키텍처 변경 없음
- 검증 비용 최소

**단점**:
- 모든 에이전트의 Jira 상태 변경 요구사항 미충족
- Notion, Design System 접근 불가
- PM 외 에이전트의 Jira 코멘트 불가

**위험도**: LOW
- 보안: Low (추가 권한 최소)
- 유지보수: Low (변경 범위 작음)
- 기능 충족: Low (요구사항 일부만 충족)

**적합 상황**: 즉각적인 1차 배포, 검증 후 확장

---

### Scenario 2: Balanced (Medium Risk) - 권장안

**전략**: 도구별 읽기/쓰기를 세분화하여 에이전트별 화이트리스트 적용

**구현**:
- 공통 읽기 도구 세트 정의 + 직군별 쓰기 도구 명시적 허용
- `transitionJiraIssue`, `addCommentToJiraIssue`는 모든 에이전트에 허용 (본인 티켓)
- PM에게 `createJiraIssue`, `editJiraIssue`, `createConfluencePage` 등 쓰기 전체 허용
- Bash 보유 에이전트에게 gh CLI, npm CLI 접근 가이드라인 추가
- WebFetch로 Notion 읽기, Storybook 참조 가능하도록 가이드

**장점**:
- 요구사항 대부분 충족
- 최소 권한 원칙 준수
- 현재 도구로 즉시 구현 가능
- 향후 MCP 추가 시 확장 용이

**단점**:
- `allowedTools` 목록이 에이전트별로 길어짐
- Notion 접근이 WebFetch 기반이라 불안정
- MCP 토큰이 공유되므로 에이전트 간 격리 불완전

**위험도**: MEDIUM
- 보안: Medium (쓰기 권한 세분화로 통제)
- 유지보수: Medium (도구 목록 관리 필요)
- 기능 충족: High (요구사항 80%+ 충족)

**적합 상황**: 현재 팀 규모와 도구 스택에 최적

---

### Scenario 3: Aggressive (High Risk) - 완전 자동화

**전략**: 커스텀 MCP 서버 개발 + 미들웨어 권한 레이어 추가

**구현**:
- Notion MCP 서버 자체 개발 (Notion API 래핑)
- Figma MCP 서버 자체 개발 (Figma REST API 래핑)
- MCP 프록시 서버를 두어 에이전트별 토큰 분리 + Rate Limiting
- 사내 npm 레지스트리 MCP 서버 개발
- Design System 컴포넌트 카탈로그 MCP 서버 개발

**장점**:
- 모든 요구사항 100% 충족
- 에이전트별 완전한 권한 격리
- 네이티브 수준의 도구 통합

**단점**:
- MCP 서버 4-5개 자체 개발 필요 (수 주 소요)
- 유지보수 부담 (외부 API 변경 시 MCP 서버도 수정)
- 과잉 엔지니어링 (현재 7명 에이전트 규모에 비해)

**위험도**: HIGH
- 보안: Low (완전 격리 가능)
- 유지보수: High (자체 MCP 서버 관리)
- 개발 비용: High (수 주 개발)
- 복잡도: High

**적합 상황**: 에이전트 팀이 프로덕션 수준으로 운영되고, 장기적 투자 여력이 있을 때

---

## 4. 권장안: Scenario 2 (Balanced)

### 선택 근거
1. **요구사항 충족률 80%+**: 핵심 요구사항(Jira/Confluence 읽기쓰기, GitHub 접근, 본인 티켓 관리)을 모두 해결
2. **즉시 구현 가능**: 추가 MCP 서버 개발 없이 현재 도구만으로 구현
3. **보안 수준 적절**: 최소 권한 원칙을 allowedTools 화이트리스트로 구현
4. **확장 가능**: 향후 Notion MCP, Figma MCP 추가 시 매핑 테이블에 행만 추가

### Scenario 1을 선택하지 않는 이유
- "모든 에이전트가 본인 Jira 티켓 상태 변경 가능" 요구사항 미충족
- 기능 충족률이 너무 낮아 실용성 부족

### Scenario 3을 선택하지 않는 이유
- 현재 에이전트 팀이 셋업/테스트 단계 (project-context.md 확인)
- 커스텀 MCP 서버 개발 비용 대비 현재 필요량이 적음
- 안정화 후 필요 시 점진적으로 전환 가능

### 확신도: HIGH
- Researcher가 모든 MCP 도구 스키마를 실제 확인함
- Security Expert의 화이트리스트 방식이 현재 아키텍처에 적합함
- PM의 우선순위 판단과 일치함

### 대비책
- Scenario 2 적용 후 Notion 접근이 빈번하다면, Scenario 3의 Notion MCP만 선별 개발

## 5. 상세 도구 매핑 테이블

### 5.1 도구 권한 분류 체계

```
[R] = 읽기 전용 (Read-only)
[W] = 쓰기 가능 (Write)
[RW] = 읽기 + 쓰기
[-] = 접근 불가
```

**GitHub 접근 규칙** (Bash + gh CLI):

- 코드 직접 수정은 보안 패치에 한정

#### Triage Agent

- Bash 사용 불가

## 8. Notion 접근 대안 (상세)

현재 Notion MCP 서버가 없으므로, WebFetch 기반 대안을 사용합니다.

### 8.1 WebFetch 경유 접근

```
WebFetch → Notion Public Page URL → HTML 파싱 → 내용 추출
```

**제약사항**:
- 공개(public) 페이지만 접근 가능
- 비공개 페이지는 Notion API 토큰이 필요 (환경 변수로 설정 후 WebFetch 헤더에 포함)
- 구조화된 데이터(데이터베이스, 관계형 속성)는 제대로 추출 불가
- 쓰기 불가

### 8.2 권장 대안: Confluence로 통합

- 핵심 문서를 Confluence에 집중
- Notion은 보조 참고용으로만 유지
- 장기적으로 Notion MCP 서버 도입 (오픈소스 존재 여부 조사 필요)

### 8.3 장기 계획: Notion MCP 서버

- npm에서 `@anthropic/mcp-server-notion` 또는 커뮤니티 MCP 서버 검색
- 존재 시: 설치 후 `.claude/settings.json`에 추가
- 미존재 시: Notion API를 래핑하는 간단한 MCP 서버 자체 개발 (1-2일 소요)

## 11. Red Flag 기준

아래 상황 발생 시 즉시 재논의:

| 상황 | 대응 |
|------|------|
| 에이전트가 타인의 Jira 이슈를 무단 변경 | 해당 에이전트의 `transitionJiraIssue` 권한 회수, persona에 규칙 강화 |
| Confluence 페이지 의도치 않은 덮어쓰기 | `updateConfluencePage` 사용 전 `getConfluencePage`로 확인하는 규칙 추가 |
| gh CLI로 비관련 레포 접근 시도 | Bash 사용 가이드라인 강화, 필요 시 allowedTools에서 Bash 패턴 제한 |
| API 호출 한도 초과 | 에이전트별 호출 빈도 모니터링, 불필요한 검색 쿼리 최적화 |
| Notion 접근 요구가 빈번 (주 10회+) | Notion MCP 서버 도입을 Phase 3에서 앞당김 |
