## 1. 시스템 아키텍처 & 설계 -- Staff

### 강점

- **Orchestrator / Coordinator / Evaluator 3자 분리**: Triage(라우팅만) / PM(조율만) / Chalmers(검증만). 대부분의 멀티에이전트 프레임워크가 못 하는 설계
- **SQLite Claim Lock**: Redis/Zookeeper 없이 `INSERT OR IGNORE` 원자성으로 단일 응답 보장. "가장 단순한 올바른 해결책"을 고른 판단력
- **3-Tier 라우팅**: @mention -> 키워드 -> LLM -> PM fallback. 비용-정확도 트레이드오프를 이해한 설계
- **선언형 경계**: 에이전트 scope의 `handles / does_not_handle`로 역할 충돌 원천 차단
- **확장성**: 새 에이전트 추가 시 persona 파일 + routing-rules 수정만으로 3-4시간 완료

### 개선 필요

| 항목 | 현재 상태 | 개선 방향 | 우선순위 |
|------|----------|----------|---------|
| `index.ts` 119K줄 | God Object -- 라우팅, 승인, QA, 압축 등 혼재 | `approval-handler.ts`, `qa-handler.ts`, `context-handler.ts`, `gateway-handler.ts`, `dispatch-handler.ts`로 분해 | **P0** |
| 전역 Map 상태 관리 | `agentSessions`, `runningTasks`, `researchModeState` 혼재 | 중앙 State Manager 도입 (EventEmitter 기반 또는 경량 상태 머신) | **P1** |

## 2. 프로세스 엔지니어링 & 워크플로우 -- Staff+

> 이 시스템의 가장 강력한 차원

### 강점

- **React Process 9단계**: Receive -> Analyze -> Dispute -> Rework -> Self-Review -> Re-verify -> Escalation -> Learning -> Hook/Skill. Reflexion, Self-Refine, AdaptUI 학술 패턴을 실무에 통합
- **Correction->Rule 파이프라인**: 실패 3회 -> Lesson 문서 -> 자가리뷰 승격 -> 정적 규칙화. 시스템이 스스로 진화하는 구조
- **동적 자가리뷰**: `qa-metrics.md`에서 본인 FAIL 빈도 -> 체크리스트 우선순위 자동 조정
- **PM HARD RULE**: 순차 워크플로 중 Slack 무음 (이모지만). 노이즈 제거에 대한 깊은 이해
- **QA 3계층**: Static(<10초) -> DB(<30초) -> Runtime(1-5분). 비용 순 실행으로 빠른 실패 보장

### 개선 필요

| 항목 | 현재 상태 | 개선 방향 | 우선순위 |
|------|----------|----------|---------|
| 프로세스 과다 명세 | 일부 규칙이 경직성 유발 가능 | 규칙별 "유연/엄격" 태그 부여, 상황 판단 여지 확보 | P2 |
| 프로세스 준수 자동 검증 | Hook 수준에 머물러 있음 | Guard Hook -> 정량 메트릭 (준수율 추적) 연동 | P2 |

## 핵심 인사이트

### 이 시스템의 차별점

1. **테스트를 에이전트화**: unit test 작성 대신 Chalmers QA 에이전트가 3-Layer 검증 수행. 더 높은 추상화
2. **학습하는 시스템**: Correction->Rule 파이프라인으로 실패가 자동으로 규칙이 됨
3. **토큰 경제학 내재화**: 200줄 persona cap, 3단계 메모리 로드, 모델 티어링 -- 모든 설계에 비용 의식
4. **조직 설계 = 시스템 설계**: Conway's Law를 역으로 활용. 원하는 조직 역학을 에이전트 아키텍처로 구현

### 유효한 기술 부채 (감점 정당)

1. **index.ts 119K줄** -- "아직 안 한" 게 아니라 쌓인 부채. 설계 원칙이 구현에 미반영
2. **전역 Map 상태 관리** -- 의도적 미루기가 아닌 구조적 문제. 버그 리스크

### 비유효 감점 (프로젝트 단계상 정상)

- CI/CD, Docker, 브랜치 전략 -> 1인 R&D 단계에서 아직 필요하지 않음
- Bloom/cost-quality 미완성 -> R&D 실험의 구조 잡기 단계
- Langfuse -> 계획이 있고 연동 지점이 코드에 존재
