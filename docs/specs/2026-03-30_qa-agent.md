# Feature Spec: QA Agent (E2E 검증 에이전트)
Date: 2026-03-30
Author: Marge (PM)
Status: approved

## 배경 & 문제

현재 검증 파이프라인:
1. 구현 에이전트 완료 → DoD 셀프 보고 (Homer 주관)
2. Cross-Verification → 보안/프론트엔드 컨벤션·규칙 체크

빠진 레이어: **스펙 AC(Acceptance Criteria)가 실제로 충족됐는지 실행으로 검증하는 단계**

- Homer의 DoD PASS는 셀프 보고 — 실제 동작 확인 아님
- Cross-Verify는 코드 품질·보안·컨벤션 검사 — 기능 작동 여부 검사 아님
- 결과: Task Queue System처럼 코드는 있지만 실제로 돌아가는지 아무도 검증 안 한 케이스 발생

## 해결책: QA Agent

Feature Spec의 AC 목록을 기준으로 **실제 실행 기반 E2E 테스트**를 수행하는 독립 에이전트.

---

## 설계 원칙

1. **AC 기준 테스트** — 스펙 파일의 AC 목록을 그대로 테스트 케이스로 사용
2. **독립 context** — 구현 에이전트 히스토리와 완전히 분리된 세션
3. **계층화 테스트** — 빠른 검증(DB/코드)부터 느린 검증(런타임)까지 계층화
4. **FAIL 시 자동 에스컬레이션** — Homer에게 실패 AC + 오류 내용 전달하여 재작업 트리거

---

## 트리거 조건

- Cross-Verification PASS 이후 자동 실행
- 또는 PM이 `run_qa` 도구로 수동 실행

---

## 구현 스펙

### 1. QA Agent 에이전트 프로필 (`.claude/agents/chalmers.md` — 기존 파일에 QA 모드 추가됨)

```
name: Chalmers (QA)
role: qa
description: E2E 테스트 에이전트. Feature Spec의 AC를 실행 기반으로 검증.
tools: Bash, Read, Glob, Grep, mcp__slack__*
```

### 2. bridge에 `run_qa` 도구 추가 (`agent-runtime.ts`)

PM이 호출하는 도구:

```typescript
tool('run_qa', {
  description: 'QA Agent를 실행하여 Feature Spec의 AC를 E2E 검증',
  inputSchema: {
    specPath: string,      // 검증할 스펙 파일 경로 (예: docs/specs/2026-03-30_task-queue-system.md)
    reason?: string        // 실행 이유 (선택)
  }
})
```

### 3. Cross-Verification 완료 후 QA 자동 트리거 (`cross-verify.ts`)

```typescript
// 모든 검증자 PASS 시
if (allPassed) {
  await runQaAgent(slackApp, event, specPath, producerResult);
}
```

### 4. QA Agent 실행 로직 (`qa-loop.ts` — Chalmers 호출 오케스트레이션)

QA Agent가 수행하는 테스트 계층:

**Layer 1 — Static Check (빠름, <10초)**
- 스펙에 명시된 파일이 존재하는지 (Glob)
- 스펙에 명시된 함수/인터페이스가 코드에 있는지 (Grep)
- DB 스키마가 스펙과 일치하는지 (Bash sqlite3)

**Layer 2 — DB State Check (중간, <30초)**
- DB 마이그레이션이 정상 적용됐는지
- 초기 상태가 올바른지 (예: running 상태 orphan 없음)

**Layer 3 — Runtime Check (느림, 1-5분)**
- 실제 Slack 메시지 발송 → 브리지 응답 확인
- DB 상태 변화 확인 (before/after)
- 에러 케이스: 의도적으로 실패 유발 → 올바른 처리 확인

### 5. QA 결과 보고 형식

```
🧪 [QA 검증 결과] Task Queue System

Layer 1 — Static Check
  ✅ queue-manager.ts 존재
  ✅ queue-processor.ts 존재
  ✅ enqueue() 함수 구현됨
  ✅ getNextTask() 함수 구현됨

Layer 2 — DB State Check
  ✅ task_queue 테이블 존재
  ✅ 필수 컬럼 (id, status, depends_on, ...) 전부 확인
  ✅ 인덱스 3개 정상 생성

Layer 3 — Runtime Check
  ✅ [AC-HP1] 3개 태스크 등록 → 큐 시작 Slack 메시지 게시됨
  ✅ [AC-HP2] Task 1 완료 → Task 2 자동 시작 (5초 이내)
  ❌ [AC-HP3] Task 2 결과가 Task 3 context로 전달됨 — FAIL
     오류: prevResult가 null로 전달됨 (queue-processor.ts:136)
  ⏭️ [AC-HP4] 완료 시간 표시 — SKIP (선행 AC 실패)

결과: 2 FAIL, 1 SKIP
→ Homer에게 재작업 위임 중...
```

### 6. FAIL 시 자동 Homer 에스컬레이션

```typescript
if (failedACs.length > 0) {
  await delegate('backend', {
    message: `QA 검증 실패. 다음 AC를 수정해주세요:\n${failedACs.join('\n')}`,
    specPath,
    failDetails,
  });
}
```

## Acceptance Criteria

### Happy Path
- [ ] Cross-Verification 전체 PASS 후 QA Agent가 자동 실행됨
- [ ] Layer 1 Static Check: 스펙 명시 파일·함수 존재 여부 검증됨
- [ ] Layer 2 DB Check: task_queue 테이블 스키마가 스펙과 일치함을 검증함
- [ ] Layer 3 Runtime Check: 실제 enqueue → 처리 → Slack 알림 흐름이 검증됨
- [ ] 결과 보고가 AC별 PASS/FAIL로 Slack에 게시됨

### 에러 케이스
- [ ] AC FAIL 시 Homer에게 실패 AC + 오류 내용이 자동 전달됨
- [ ] Layer 3 실행 중 브리지 무응답(타임아웃) 시 SKIP 처리됨 (FAIL 아님)
- [ ] 스펙 파일 없을 시 명확한 오류 메시지 반환

### 엣지 케이스
- [ ] Layer 1 FAIL 시 Layer 2/3는 실행하지 않고 즉시 에스컬레이션
- [ ] 이미 QA 실행 중인 스펙에 대해 중복 실행 방지
- [ ] QA 실행 중 ⛔ 리액션 시 진행 중단
