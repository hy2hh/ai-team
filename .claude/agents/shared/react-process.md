# React Process — 피드백 대응 프로토콜

> 모든 에이전트가 피드백(QA FAIL, REQUEST_CHANGES, 디자인 위반, 재작업 요청, 보안 이슈)을
> 받았을 때 따르는 표준 절차. 리서치 기반: Self-Refine, Reflexion, Windsurf Correction→Rule.

## 1. 피드백 수신 (Receive) `[Hook 후보]`

### 수신 확인 의무
- 피드백 수신 즉시 Slack 이모지 리액션 (👀) + "피드백 확인, 검토 시작합니다" 응답
- 수신 확인 없이 작업 시작 금지

### 피드백 유형 분류

| 유형 | 발신자 | 우선순위 |
|------|--------|----------|
| QA FAIL | Chalmers | Critical → 즉시 |
| REQUEST_CHANGES | 코드리뷰어 | Severity에 따름 |
| 디자인 위반 | Krusty | Important |
| 재작업 요청 | Marge | PM 지정 우선순위 |
| 보안 이슈 | Wiggum | Critical → 즉시 |

## 2. 분석 (Analyze) — 도구 기반 독립 검증 `[Skill 후보]`

> Reflexion 패턴: 감이 아닌 **외부 데이터로 비판을 검증**한다.

각 피드백 항목에 대해:
1. **재현 확인**: 도구(Grep/Read/Bash)로 이슈가 실제 존재하는지 독립 확인 — 피드백을 맹신하지 않음
2. **원인 분류**: 내 실수 / 요구사항 오해 / 외부 의존성 / 피드백 오류
3. **영향 범위**: 수정이 다른 기능에 미치는 사이드 이펙트 평가

## 3. 이의 제기 (Dispute) — 선택적 `[Skill 후보]`

### 이의 제기가 가능한 경우
- 피드백이 사실과 다름 (코드 증거로 반박 가능)
- 피드백이 기존 아키텍처 결정과 충돌
- 피드백 수용 시 다른 AC가 깨짐

### 이의 제기 형식
```
🔄 이의 제기 — [{피드백 항목 번호}]

**피드백 내용**: [원문 인용]
**반박 근거**: [코드 증거 — 파일:라인 참조 필수]
**제안**: [대안 제시]
**영향**: [수용 시 vs 미수용 시 비교]
```

### 이의 제기 절차
1. Slack에 이의 제기 게시 (발신자 @mention)
2. 발신자 24시간 내 응답:
   - ACCEPT → 해당 항목 제외 후 재작업
   - REJECT + 근거 → 원래대로 수정
   - ESCALATE → PM(Marge)에게 판정 위임
3. 무응답 24시간 → PM 자동 에스컬레이션

### QA FAIL 이의 제기 특칙
- Chalmers FAIL에 이의 시 PM(Marge)이 중재자
- Marge는 "AC 기준 해석"에 대해서만 판정
- 기술적 쟁점은 도메인 전문가(Homer/Bart/Wiggum)에게 의견 요청

## 4. 재작업 (Rework) — Self-Refine 포함 `[Skill 후보]`

> Self-Refine 패턴: 재검증 요청 전에 **자기 비판 루프**를 돌린다.

### 재작업 절차
1. 피드백 항목을 체크리스트로 변환
2. 항목별 수정
3. **Self-Refine**: "이 수정이 피드백을 충분히 해결했는가?" 자문 → 부족하면 재수정
4. 자가 리뷰 (§5) 통과 확인
5. 재작업 보고: `[Hook 후보]`

```
🔧 재작업 완료 — [{원본 피드백 참조}]

수정 항목:
- [x] {항목1}: {수정 내용} (파일:라인)
- [x] {항목2}: {수정 내용} (파일:라인)

Self-Refine: 자기 비판 통과
자가 리뷰: 통과 (역할별 체크리스트 기준)
```

### 재작업 횟수 제한
- 같은 이슈 3회 재작업 실패 → 도메인별 에스컬레이션 (§7)

## 5. 자가 리뷰 (Self-Review) — 전 직군 의무 `[Hook 후보]`

### 공통 자가 리뷰 (모든 에이전트)
- [ ] 완료 조건(AC) 전부 대조했는가
- [ ] 수정 사항이 다른 기능을 깨뜨리지 않는가
- [ ] 이전 피드백에서 지적된 동일 패턴을 반복하지 않았는가 (학습 루프 참조)

### 역할별 추가 체크리스트
각 에이전트 페르소나 파일의 "자가 리뷰" 섹션에 정의

### 동적 우선순위 (AdaptUI 패턴)
- 세션 시작 시 `.memory/facts/qa-metrics.md`에서 자기 역할의 FAIL 빈도 확인
- 자주 실패하는 항목을 우선 체크
- `.memory/decisions/lesson-*` 파일에서 자기 역할 관련 항목 로드 → 임시 체크리스트에 추가

## 6. 재검증 (Re-verify)

| 원본 피드백 유형 | 재검증 담당 | 방법 |
|-----------------|------------|------|
| QA FAIL | Chalmers | 실패 AC만 재검증 |
| REQUEST_CHANGES | 원본 리뷰어 | 수정 항목만 확인 |
| 디자인 위반 | Krusty | 수정 UI 스크린샷 확인 |
| 보안 이슈 | Wiggum | 수정 코드 보안 리뷰 |

### 재검증 요청 형식
```
🔍 재검증 요청 — @{검증자}

원본 피드백: [{링크 또는 요약}]
수정 내역: [{항목별 파일:라인}]
```

## 8. 학습 루프 (Adaptive Evolution)

> Windsurf Correction→Rule + Cline 자동 발견 + BabyAGI 스킬 자동 생성 패턴 적용.

### 8-1. 실패 학습 (Correction → Rule)
- 동일 유형 피드백 3회 이상 → `.memory/decisions/YYYY-MM-DD_lesson-{topic}.md` 작성 `[Hook 후보]`
- lesson의 "예방 체크" 항목 → 해당 에이전트 자가 리뷰에 자동 승격 후보
- 3회 이상 유효한 lesson 항목 → 페르소나 파일 정적 체크리스트로 영구 승격 제안

### 8-2. 성공 학습 (패턴 → 스킬 후보)
- 동일 디버깅/수정 패턴 3회 이상 성공 → `.memory/decisions/YYYY-MM-DD_skill-candidate-{topic}.md`
- PM이 스킬 후보 검토 → 유효하면 `.claude/skills/`로 승격

### 8-3. QA 메트릭 누적 (Accept/Reject 추적)
- Chalmers가 매 QA 후 `.memory/facts/qa-metrics.md`에 기록:
  - 에이전트명 / 날짜 / 판정(PASS/FAIL) / FAIL 카테고리
- 이 데이터로: 약점 가시화, 자가 리뷰 우선순위 조정, 학습 루프 트리거

### 8-4. Chalmers 반복 FAIL 분석 → 규칙 개선 제안
- 매 10회 QA 후 반복 FAIL 패턴 분석
- 특정 에이전트의 특정 카테고리 반복 FAIL → 자가 리뷰 항목 추가 제안
- 공유 규칙의 모호한 부분이 반복 오해 유발 → 규칙 명확화 제안을 PM에게 보고

### 8-5. 라우팅 적응 (Triage 전용)
- 라우팅 오류 피드백 누적 → 키워드 테이블 조정 제안
- 오류 3회 누적 시 PM에게 routing-rules.md 수정 제안

### 학습 파일 형식
```markdown
# Lesson: {반복 이슈 요약}
- 발생 횟수: {N}회
- 최근 발생: {날짜}
- 근본 원인: {분석}
- 예방 체크: {자가 리뷰에 추가할 항목}
- 진화 분류: What={대상} / When={시점} / How={방법}
```

## 9. Hook/Skill 분류 요약

> paddo.dev 패턴: 결정론적 규칙은 Hook, 판단 필요한 규칙은 Skill로 분리.

| 규칙 | 분류 | 이유 |
|------|------|------|
| 수신 확인 (§1) | Hook 후보 | 100% 필수, 판단 불필요 |
| 재작업 보고 형식 (§4) | Hook 후보 | 형식 고정, 자동 검증 가능 |
| 자가 리뷰 실행 (§5) | Hook 후보 | 완료 선언 전 필수 게이트 |
| lesson 파일 작성 (§8) | Hook 후보 | 3회 임계치 자동 감지 가능 |
| 분석/재현 확인 (§2) | Skill 후보 | 도구 선택에 판단 필요 |
| 이의 제기 판단 (§3) | Skill 후보 | 기술적 판단 필요 |
| Self-Refine 자문 (§4) | Skill 후보 | 충분성 판단 필요 |
| 영향 범위 평가 (§2) | Skill 후보 | 맥락 의존적 판단 |
