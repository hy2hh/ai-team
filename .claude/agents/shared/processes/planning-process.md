# Planning Process

> **Iron Law**: 설계 없이 구현하지 않는다. 모든 프로젝트는 이 프로세스를 거친다.

## 원칙

"이건 너무 간단해서 설계가 필요 없다"는 가장 위험한 가정이다. 간단한 프로젝트야말로 검토 없는 가정이 가장 많은 낭비를 만든다. PM Donald가 주도하고, 관련 에이전트가 참여한다.

---

## Part 1: Brainstorming (PM 주도)

### HARD GATE
구현 행위(코드 작성, 파일 생성, 환경 설정)는 설계 승인 전에 **절대 시작하지 않는다**.

### 9단계 브레인스토밍

#### 1. 프로젝트 맥락 탐색
- `.memory/facts/project-context.md` 읽기
- 기존 코드, 문서, 최근 커밋 확인
- 관련 `.memory/decisions/` 확인

#### 2. 관련 에이전트 소집
- PM Donald가 Slack #ai-team에서 주제와 참여 요청 에이전트를 명시
- 각 에이전트가 자신의 관점에서 초기 의견 제공

#### 3. 명확화 질문
- PM Donald가 **한 번에 하나의 질문**을 한다
- 가능하면 다지선다형으로 제시
- 관련 에이전트가 Slack 스레드에서 답변

#### 4. 관점별 입력 수집
| 에이전트 | 제공하는 관점 |
|---------|-------------|
| @Designer Donald | 비주얼/UX 관점, 사용자 흐름 |
| @Researcher Donald | 시장 데이터, 경쟁사 분석, 트렌드 |
| @Frontend Donald | 프론트엔드 기술 제약, 구현 복잡도 |
| @Backend Donald | 시스템 아키텍처, 데이터 모델, API 설계 |
| @SecOps Donald | 보안 요구사항, 위협 모델, 컴플라이언스 |

#### 5. 접근 방식 제안 (2-3가지)
- PM Donald가 수집된 의견을 바탕으로 2-3가지 접근 방식을 정리
- 각 방식의 장단점(trade-off)을 명시
- sid에게 최종 방향 승인 요청

#### 6. 설계 문서 초안 작성
- PM Donald가 승인된 방향으로 설계 초안 작성
- `.memory/decisions/YYYY-MM-DD_{topic}.md`에 기록

#### 7. 기술 검증 루프
- @Backend Donald, @Frontend Donald가 기술적 실현 가능성 검증
- @SecOps Donald가 보안 관점 검증
- 이슈 발견 시 → PM Donald가 설계 수정 → 재검증 (최대 3회)

#### 8. sid 최종 리뷰
- 검증된 설계를 sid에게 보고
- sid 승인 후에만 구현 단계로 진행

#### 9. 구현 계획으로 전환
- Part 2: Writing Plans로 이동
- 설계 없이 구현으로 직접 넘어가지 않는다

## Part 2: Writing Plans (PM 주도)

### 계획서 작성 원칙

**가정**: 구현자는 유능하지만 이 프로젝트의 맥락을 모른다. 모든 것을 문서화한다:
- 어떤 파일을 만들거나 수정하는지
- 구체적 코드 (설명이 아닌 실제 코드)
- 어떻게 테스트하는지
- 참조할 문서

### Bite-Sized Task 분해

각 Task는 **2-5분 단위**의 단일 행동이다:

```
❌ "사용자 인증 시스템 구현"
✅ Task 1: 실패 테스트 작성 (POST /auth/login → 200)
   Task 2: 테스트 실행하여 실패 확인
   Task 3: 로그인 엔드포인트 최소 구현
   Task 4: 테스트 실행하여 통과 확인
   Task 5: 커밋
```

### Task 구조

각 Task에 반드시 포함:
- **담당 에이전트**: 누가 이 Task를 수행하는가
- **Files**: 생성/수정/테스트할 정확한 파일 경로
- **Steps**: 체크박스 목록으로 된 구체적 단계
- **Code**: 설명이 아닌 실제 구현 코드
- **Verification**: 완료 확인 방법 (명령어 + 예상 출력)

### 계획서 형식

`shared/templates/implementation-plan.md` 템플릿을 사용한다.

### 계획 리뷰 루프

1. PM Donald가 계획 초안 작성
2. 관련 에이전트가 Slack에서 리뷰 (기술적 실현 가능성, 빠진 부분, 추정 정확도)
3. 이슈 발견 시 → 수정 → 재리뷰 (최대 3회)
4. 3회 초과 시 → sid에게 에스컬레이션

## YAGNI 원칙

- "나중에 필요할 수 있으니까" → 지금은 구현하지 않는다
- "더 확장 가능하게" → 현재 요구사항만 충족하는 최소한으로
- "제대로 하자" → 현재 문제를 해결하는 가장 단순한 방법으로

---

## Red Flags

- ❌ 설계 승인 없이 코드 작성 시작
- ❌ "이건 간단하니까 바로 구현하자"
- ❌ 한 명의 관점만으로 설계 완료
- ❌ 기술 검증 없이 계획서 확정
- ❌ "나중에 필요하니까 미리 만들자" (YAGNI 위반)
- ❌ Task가 너무 큼 (5분 초과)
