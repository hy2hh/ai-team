# Code Review Protocol

> **Iron Law**: 리뷰 없는 코드는 프로덕션에 들어가지 않는다. 성과주의 합의 금지.

## 원칙

코드 리뷰는 품질 게이트이자 팀 학습의 핵심 메커니즘이다. 리뷰를 요청하는 것도, 받는 것도 체계적 프로세스를 따른다. "Great point!"이나 "Thanks for catching that!" 같은 성과주의적 반응은 금지 — 기술적 내용에만 집중한다.

---

## Part 1: 리뷰 요청 (Requesting)

### 언제 리뷰를 요청하는가

**필수:**
- 주요 기능 구현 완료 후
- main 브랜치 병합 전
- 보안에 영향을 주는 변경

**권장:**
- 막혔을 때
- 리팩토링 전/후
- 복잡한 버그 수정 후

### 리뷰 요청 시 필수 정보

`shared/templates/code-review-request.md` 템플릿을 사용하여 Slack에 리뷰를 요청한다:

1. **What**: 무엇을 구현/변경했는가
2. **Why**: 왜 이 변경이 필요한가
3. **Context**: 리뷰어가 알아야 할 배경
4. **Files Changed**: 변경된 파일 목록과 각 파일의 변경 이유
5. **Review Focus**: 특히 봐달라는 부분

### 리뷰어 매트릭스

| 리뷰 대상 | 주 리뷰어 | 보조 리뷰어 | 리뷰 관점 |
|-----------|----------|-----------|----------|
| Frontend 코드 | @Homer | @Krusty | API 계약 준수 / UI 구현 충실도 |
| Backend 코드 | @Wiggum | @Bart | 보안 / API 소비자 관점 |
| API 계약 변경 | @Bart + @Homer | @Wiggum | 양측 호환성 / 보안 |
| 보안 관련 코드 | @Wiggum | @Homer | 취약점 / 아키텍처 |
| UI 구현 | @Krusty | @Bart | 디자인 스펙 일치 / 접근성 |
| Frontend 디자인 전용 변경 | @Krusty | — | 디자인 스펙 일치 |

> **디자인 전용 프론트 변경 판정 기준**: CSS/스타일링 변경만 포함되고 API 호출·비즈니스 로직·컴포넌트 구조 변경이 없는 경우. 이 경우 리뷰어는 @Krusty(Designer)만 포함하고, @Homer(Backend)·@Chalmers(QA) 제외. `recommend_next_phase`에서 `hasCodeChanges: false` 설정. 코드 로직 변경이 1줄이라도 있으면 이 규칙 적용 불가.

## Part 2: 리뷰 응답 (Receiving)

### 6단계 응답 프로세스

리뷰 피드백을 받으면 반드시 이 순서를 따른다:

#### 1. READ — 전체 피드백 읽기
- 반응하지 않고 **전체** 피드백을 먼저 읽는다
- 항목들이 서로 연관될 수 있으므로 부분 이해로 행동하지 않는다

#### 2. UNDERSTAND — 요구사항 재진술
- 각 피드백 항목의 기술적 요구사항을 자신의 말로 재진술한다
- 불명확한 항목은 **구현 전에** 질문한다

#### 3. VERIFY — 코드베이스에서 확인
- 리뷰어가 지적한 내용이 실제로 코드베이스에서 어떻게 되어 있는지 확인
- "맞을 것이다"로 넘어가지 않는다

#### 4. EVALUATE — 기술적 타당성 평가
- 이 피드백이 **이 코드베이스**에서 기술적으로 올바른가?
- YAGNI 체크: 리뷰어가 "제대로 구현하자"고 제안하면, 실제로 그 기능이 사용되는지 확인

#### 5. RESPOND — 기술적 응답
- ✅ 올바른 응답: "수정함. [간단한 설명]" / "확인 결과 [X]이므로 [Y]로 변경"
- ✅ 반박 시: "현재 구현의 이유: [기술적 근거]. [대안]이 더 나은 이유를 설명해주시겠습니까?"
- ❌ 금지: "You're absolutely right!", "Great point!", "Thanks for catching that!"
- ❌ 금지: 검증 전에 "바로 구현하겠습니다"

#### 6. IMPLEMENT — 하나씩 수정
- 한 번에 하나의 항목을 수정하고 테스트
- 각 수정 후 검증 (verification-before-completion 적용)

### 반박이 적절한 경우

다음 경우에는 기술적 근거와 함께 반박한다:
- 기존 기능이 깨지는 경우
- 리뷰어가 맥락을 모르는 경우
- YAGNI 위반인 경우
- 기술적으로 틀린 경우
- 아키텍처 결정과 충돌하는 경우

### YAGNI 체크

리뷰어가 "이건 나중에 필요하니까 미리 구현하자"고 제안하면:
1. 코드베이스에서 실제 사용처를 검색
2. 사용되지 않으면 → "현재 사용처가 없어 YAGNI 원칙에 따라 제거/미구현합니다"
3. 사용되면 → 구현

## Severity 분류

| Severity | 의미 | 행동 |
|----------|------|------|
| **Critical** | 보안 취약점, 데이터 손실 위험, 기능 장애 | 즉시 수정. 병합 차단. |
| **Important** | 성능 문제, 아키텍처 위반, 테스트 누락 | 다음 진행 전 수정 |
| **Minor** | 네이밍, 스타일, 문서화 | 기록하고 현재 또는 다음 작업에서 수정 |

## Red Flags

- ❌ 리뷰 없이 병합
- ❌ 모든 피드백에 무조건 동의 (비판적 사고 부재)
- ❌ 피드백을 무시하고 진행
- ❌ 성과주의적 응답 ("Excellent feedback!", "You're so right!")
- ❌ 리뷰 요청 시 컨텍스트 없이 "봐주세요"
- ❌ Spec 리뷰 전에 Code Quality 리뷰 진행

## 리뷰 순서

1. **Spec Compliance Review** — 요구사항을 충족하는가? (기능적 정확성)
2. **Code Quality Review** — 코드 품질이 기준을 만족하는가? (구조, 테스트, 보안)

Spec 리뷰가 통과하지 않으면 Code Quality 리뷰를 시작하지 않는다.
