# Definition of Done (DoD)

> **Iron Law**: "동작하면 완료"가 아니라 "기준을 충족하면 완료".

## 왜 필요한가

기준 없이 구현하면 happy path가 동작하는 순간 완료로 간주한다.
에러 처리, 접근성, 하드코딩 같은 문제는 "실력 부족"이 아니라 "기준 부재"에서 온다.

---

## 공통 DoD (모든 구현 작업)

| # | 항목 | 설명 |
|---|------|------|
| 1 | 에러 핸들링 | 모든 async 함수에 try-catch. 사용자에게 의미 있는 에러 메시지 표시 |
| 2 | 하드코딩 금지 | 매직 넘버/문자열 → 상수 또는 환경변수로 분리 |
| 3 | AC 전체 통과 | 스펙의 Happy Path + 에러 케이스 + 엣지 케이스 AC 전부 충족 |
| 4 | 빌드 성공 | `tsc --noEmit` + `pnpm lint` 에러 없음 |
| 5 | 런타임 테스트 | 실제 실행하여 동작 확인 (타입 체크만으로 불충분) |

## Frontend 추가 DoD

| # | 항목 | 설명 |
|---|------|------|
| 6 | 접근성 | 인터랙티브 요소에 aria-label, 키보드 네비게이션 가능 |
| 7 | 로딩/에러 UI | 로딩 스피너, 에러 상태, 빈 상태 모두 처리 |
| 8 | 반응형 | 모바일 뷰포트에서 레이아웃 깨지지 않음 |

## Backend 추가 DoD

| # | 항목 | 설명 |
|---|------|------|
| 6 | 입력 검증 | 모든 엔드포인트에서 요청 바디/파라미터 유효성 검증 |
| 7 | 에러 응답 | 일관된 에러 응답 형식 (status code + error message) |
| 8 | N+1 방지 | 루프 내 쿼리 금지, JOIN 또는 배치 쿼리 사용 |

## Designer 추가 DoD

| # | 항목 | 설명 |
|---|------|------|

| 7 | 인터랙션 스펙 | hover, active, disabled, focus 상태 정의 |

---

## 완료 보고 시 필수 포함

```
@Marge 작업 완료했습니다.

DoD 체크:
- [x] 에러 핸들링: 모든 API 호출에 try-catch + 에러 UI
- [x] 하드코딩: 상수 분리 완료 (BOARD_ID, API_URL)
- [x] AC: Happy Path 3/3, 에러 케이스 2/2, 엣지 케이스 1/1
- [x] 빌드: tsc + lint 통과
- [x] 런타임: 실행 확인 (스크린샷/로그 첨부)
```

## 이 기준은 협상 대상이 아니다

"간단한 작업이라 DoD 생략" 없음. 간단한 작업이면 DoD 통과도 간단하다.
