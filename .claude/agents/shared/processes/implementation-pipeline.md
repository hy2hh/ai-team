# Implementation Pipeline

> **Iron Law**: 계획 → 구현 → 자가 리뷰 → Spec 리뷰 → Code Quality 리뷰. 순서를 건너뛰지 않는다.

## 원칙

Marge가 작성한 구현 계획(`shared/templates/implementation-plan.md`)을 기반으로, 각 Task를 담당 에이전트에 할당하고 3단계 검증을 거쳐 완료한다. 구현 품질은 체계적 파이프라인으로 보장한다.

---

## Task 할당 패턴

### Marge의 역할
1. 구현 계획서의 각 Task를 Slack #ai-team에서 담당 에이전트에 할당
2. Task의 전체 맥락을 메시지에 포함 (Task 본문 + 관련 설계 결정 + 의존성)
3. 독립적인 Task는 병렬로 할당 가능
4. 의존성이 있는 Task는 선행 Task 완료 후 순차 할당

### 할당 메시지 포맷
```
@[에이전트명] Task [번호] 할당합니다.

[Task 전체 내용 — 계획서에서 복사]

의존성: [선행 Task 완료 여부]
맥락: [이 Task를 이해하는 데 필요한 추가 정보]
```

**중요**: Task 내용을 "계획서 참조"로 대체하지 않는다. 전체 내용을 메시지에 포함한다.

---

## 3단계 검증 파이프라인

모든 Task는 이 3단계를 **순서대로** 통과해야 한다:

### Stage 1: 자가 리뷰 (구현 에이전트)

구현 완료 후, 담당 에이전트가 스스로 다음을 확인:

**공통 체크리스트:**
- [ ] 계획서의 Steps를 모두 완료했는가?
- [ ] 테스트가 통과하는가?
- [ ] 빌드가 성공하는가?
- [ ] 기존 기능이 깨지지 않았는가? (회귀 테스트)
- [ ] 코드가 의도한 대로 동작하는가? (수동 확인)

### Stage 2: Spec Compliance Review (다른 에이전트)

구현이 **요구사항을 정확히 충족**하는지 확인:

- 계획서의 Verification 항목이 모두 통과하는가?
- 누락된 요구사항이 없는가?
- 엣지 케이스가 처리되었는가?
- 리뷰어: `code-review-protocol.md`의 리뷰어 매트릭스 참조

### Stage 3: Code Quality Review (다른 에이전트)

코드 **품질**이 기준을 충족하는지 확인:

- 코드 구조와 가독성
- 에러 처리와 엣지 케이스
- 보안 (입력 검증, 인증/인가)
- 성능 (불필요한 연산, 메모리 사용)
- 테스트 커버리지와 품질

**순서 중요**: Spec 리뷰가 통과하지 않으면 Code Quality 리뷰를 시작하지 않는다.

## 상태 보고

구현 에이전트는 Task 완료 시 다음 상태 중 하나를 보고한다:

| 상태 | 의미 | 다음 행동 |
|------|------|----------|
| **DONE** | 모든 요구사항 충족, 자가 리뷰 통과 | Stage 2로 진행 |
| **DONE_WITH_CONCERNS** | 완료했지만 우려 사항 있음 | 우려 내용 명시 → Marge 판단 |
| **BLOCKED** | 진행 불가 | 차단 원인 명시 → Marge가 해결 |
| **NEEDS_CONTEXT** | 추가 정보 필요 | 필요한 정보 명시 → Marge 또는 다른 에이전트 제공 |

### BLOCKED 처리

| 차단 원인 | 해결 방법 |
|-----------|----------|
| 맥락 부족 | Marge가 추가 정보 제공 |
| 기술적 한계 | 아키텍처 논의 → 필요시 sid 에스컬레이션 |
| Task 너무 큼 | Marge가 Task 분해 |
| 계획 오류 | Marge가 계획 수정 → 관련 에이전트 재검증 |

## 병렬 구현 패턴

독립적인 Task는 서로 다른 에이전트에게 동시에 할당할 수 있다:

```
Task 1 (Backend) ─────→ 완료 ─→ Stage 2, 3
                                              ↘
Task 2 (Frontend) ────→ 완료 ─→ Stage 2, 3    → 통합 테스트
                                              ↗
Task 3 (Frontend) ────→ 완료 ─→ Stage 2, 3
```

**주의**: 동일 에이전트에게 여러 Task를 동시 할당하지 않는다. 한 Task 완료 후 다음 Task를 할당한다.

---

## 전체 구현 완료 후

모든 Task가 3단계 검증을 통과하면:

1. **통합 테스트**: 전체 시스템이 함께 동작하는지 확인
2. **최종 코드 리뷰**: 전체 구현에 대한 최종 리뷰 (`code-review-protocol.md` 적용)
3. **완료 보고**: Marge가 sid에게 구현 완료 보고 (verification-before-completion 적용)

---

## Red Flags

- ❌ 계획 없이 구현 시작
- ❌ Spec 리뷰 전에 Code Quality 리뷰 진행
- ❌ 자가 리뷰 건너뛰기
- ❌ "계획서 참조"로 Task 내용 대체 (전체 맥락을 제공해야 함)
- ❌ 동일 에이전트에게 동시에 여러 Task 할당
- ❌ BLOCKED 상태를 무시하고 진행
- ❌ 미수정 이슈가 있는 상태에서 다음 Task 진행
- ❌ "대충 맞으니까"로 Spec 리뷰 통과 처리
