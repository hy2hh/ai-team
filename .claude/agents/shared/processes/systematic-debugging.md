# Systematic Debugging

> **Iron Law**: 수정 시도 전에 반드시 근본 원인을 찾아라. 증상 수정은 실패다.

## 원칙

버그, 테스트 실패, 예상치 못한 동작이 발생하면 즉시 수정을 시도하지 말고, 체계적으로 근본 원인을 추적한다. "빠른 수정"은 거의 항상 새로운 문제를 만든다.

---

## Phase 1: Root Cause Investigation

### 1.1 에러 메시지 완전 읽기
- 스택 트레이스를 **끝까지** 읽는다
- 에러 메시지의 모든 정보를 활용한다 (파일명, 라인 번호, 변수값)
- "대충 아는 것 같다"로 넘어가지 않는다

### 1.2 재현 확인
- 문제를 **일관되게** 재현할 수 있는가?
- 재현 조건을 정확히 기록한다 (입력값, 환경, 순서)
- 간헐적 문제 → 재현 조건을 좁히는 것이 최우선

### 1.3 최근 변경 확인
- `git diff`, `git log`로 최근 변경사항 확인
- "언제부터 문제가 발생했는가?"를 파악
- 변경 없이 갑자기 발생 → 외부 의존성, 환경 변화 확인

### 1.4 데이터 흐름 추적 (Root Cause Tracing)
- 에러가 **나타나는 곳**이 아니라 **시작되는 곳**을 찾는다
- 역방향 추적: 에러 → 직접 원인 → 호출자 → 최초 트리거
- 각 단계에서 데이터가 어떻게 변환되는지 확인

```
증상 관찰
  ↓
직접 원인 찾기 (이 코드가 에러를 발생시킴)
  ↓
"이것을 호출한 것은?" (콜 체인 역추적)
  ↓
최초 트리거 발견 (잘못된 데이터가 시작된 곳)
  ↓
여기를 수정 ← 이것이 근본 원인
```

### 1.5 팀 협업 시나리오
- 다른 에이전트의 작업 결과가 원인일 수 있는 경우:
  1. 해당 에이전트의 최근 변경 사항을 `.memory/`에서 확인
  2. Slack에서 @mention하여 변경 의도와 맥락 질문
  3. 공동으로 문제 범위를 좁힘
  4. 원인이 확인되면 담당 에이전트가 수정

## Phase 2: Pattern Analysis

### 2.1 동작하는 예시 찾기
- 같은 기능이 정상 동작하는 곳이 있는가?
- 정상 동작 코드와 문제 코드의 차이점을 식별

### 2.2 참조 코드 비교
- 공식 문서, 예제 코드와 비교
- 참조 코드를 **완전히** 읽는다 — 부분만 보지 않는다

### 2.3 차이점 식별
- 정상 vs 비정상의 구체적 차이점을 목록화
- 환경 차이, 설정 차이, 의존성 버전 차이 확인

### 2.4 의존성 파악
- 문제 코드가 의존하는 모든 요소를 매핑
- 숨겨진 의존성 (전역 상태, 환경 변수, 실행 순서) 확인

---

## Phase 3: Hypothesis Testing

### 3.1 단일 가설 수립
- 한 번에 **하나의** 가설만 세운다
- 가설을 명확하게 서술: "X 때문에 Y가 발생한다"

### 3.2 최소한의 테스트
- **한 번에 하나의 변수만** 변경하여 테스트
- 여러 것을 동시에 바꾸지 않는다

### 3.3 결과 검증
- 가설이 맞았는가? 증거는?
- 가설이 틀렸다면 → 새로운 증거를 기반으로 다음 가설

### 3.4 불확실할 때
- "잘 모르겠다"를 인정하고 조사한다
- 공식 문서, 소스 코드를 확인한다
- 추측으로 코드를 수정하지 않는다

---

## Phase 4: Implementation

### 4.1 실패 테스트 먼저
- 수정 전에 **실패하는 테스트 케이스**를 작성한다
- 이 테스트가 버그를 정확히 재현하는지 확인

### 4.2 단일 수정
- 근본 원인에 대한 **단일 수정**을 구현한다
- 여러 문제를 한 번에 수정하려 하지 않는다

### 4.3 수정 검증
- 실패 테스트가 이제 통과하는지 확인
- 기존 테스트가 여전히 통과하는지 확인 (회귀 테스트)

### 4.4 방어적 계층 추가 (Defense in Depth)
- 수정 후, 같은 유형의 버그가 **구조적으로 불가능**하도록 방어:
  - Layer 1: 입력 지점에서 유효성 검사
  - Layer 2: 비즈니스 로직에서 검증
  - Layer 3: 환경 가드 (위험한 작업 차단)
  - Layer 4: 디버그 계측 (에러 컨텍스트 기록)

### 4.5 수정 실패 카운터 — 3회 규칙
- 수정 시도 횟수를 추적한다
- **3회 실패 시**: 더 이상 같은 접근을 반복하지 않는다
  1. 현재까지의 시도와 결과를 정리
  2. sid에게 에스컬레이션 + Slack에서 아키텍처 토론 스레드 시작
  3. "가설이 틀렸다"가 아니라 **"접근 자체가 틀렸다"**를 의심

## Condition-Based Waiting (테스트 안정성)

타이밍 관련 문제에서 임의의 `setTimeout`이나 `sleep` 대신 조건 기반 대기를 사용한다:

```typescript
// ❌ 추측 기반 대기
await new Promise((r) => setTimeout(r, 50));

// ✅ 조건 기반 대기
await waitFor(() => getResult() !== undefined);
```

- 항상 타임아웃을 포함한다
- 폴링 간격은 10ms가 적절
- 조건 함수 안에서 최신 값을 조회한다

이런 생각이 들면 Phase 1을 건너뛰려는 것이다:

| 핑계 | 현실 |
|------|------|
| "일단 빠르게 고치자, 나중에 조사" | Phase 1부터 해라 |
| "완전히 이해 못하지만 될 수도" | 이해 없는 수정 = 새 버그 |
| "한 번 더 시도" (2+ 실패 후) | 접근 자체에 의문 제기해라 |
| "간단한 문제니까 바로 수정" | 간단해 보이는 곳에서 가장 큰 오류 |
| "여러 개 동시에 바꾸면 더 빠르다" | 한 번에 하나, 검증 후 다음 |
| "이 패턴이 맞는데 다르게 적용" | 패턴을 따르든지 새 가설을 세우든지 |

## Red Flags — 이 신호가 보이면 멈춰라

- ❌ "일단 빠르게 고치고 나중에 조사하자"
- ❌ "그냥 X를 바꿔보자"
- ❌ "테스트는 건너뛰자"
- ❌ "완전히 이해하진 못하지만 이게 될 수도..."
- ❌ 데이터 흐름 추적 전에 수정안 제시
- ❌ 아키텍처 논의 없이 3회 이상 수정 시도

## 에스컬레이션 트리거

| 상황 | 행동 |
|------|------|
| 3+ 수정 실패 | sid에게 에스컬레이션 + 아키텍처 토론 |
| 다른 에이전트 작업이 원인 | 해당 에이전트 @mention + 공동 디버깅 |
| 재현 불가 | 디버그 계측 추가 + 모니터링 강화 요청 |
| 근본 원인이 외부 의존성 | sid에게 보고 + 우회 방안 제시 |
