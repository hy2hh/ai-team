# Task Delegation Template

PM이 구현 에이전트에게 Task를 위임할 때 이 구조를 따른다.

---

## 위임 메시지 필수 구조

```
@[에이전트명] [Task 제목] 위임합니다.

### Task Description
[계획서에서 해당 Task의 전체 텍스트를 복사. "계획서 참조" 금지.]

### Context
- **전체 흐름에서의 위치**: 이 Task는 [기능명]의 N번째 단계
- **선행 의존성**: [Task X 완료됨 / 없음]
- **후행 작업**: 이 Task 완료 후 [다음 Task/리뷰]로 진행

### Files
- Create: `정확한/파일/경로.ts`
- Modify: `정확한/파일/경로.ts:라인범위`

### Verification
\```bash
[완료 확인 명령어]
# 예상 출력: [구체적 출력]
\```

### Before You Begin
불명확한 점이 있으면 **코드 작성 전에 이 스레드에서 질문하라.**
추측으로 진행하지 말 것.

### Escalation
막히면 멈추고 보고하라. 페널티 없음. 나쁜 작업이 무작업보다 나쁘다.
```

## 수신 에이전트 응답 프로토콜

### 상태 코드 (보고 시 반드시 사용)

| 상태 | 의미 | 다음 행동 |
|------|------|---------|
| **DONE** | 모든 요구사항 충족 | Stage 2 (Spec Review)로 진행 |
| **DONE_WITH_CONCERNS** | 완료했지만 우려 있음 | Marge가 우려 평가 후 판단 |
| **BLOCKED** | 진행 불가 | Marge가 블로커 해결 |
| **NEEDS_CONTEXT** | 추가 정보 필요 | Marge/다른 에이전트가 제공 |

### Self-Review (보고 전 필수)
- [ ] 요구사항 전부 구현했는가? 빠진 것 없는가?
- [ ] 테스트가 실제 동작을 검증하는가?
- [ ] YAGNI — 요청된 것만 빌드했는가?
- [ ] 기존 코드베이스 패턴을 따랐는가?

이슈 발견 시 보고 전에 스스로 수정한다.
