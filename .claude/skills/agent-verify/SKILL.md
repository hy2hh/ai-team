# Verification Before Completion + Definition of Done

> **Iron Law**: 완료 주장 전에 반드시 검증 증거를 확보하라. 예외 없음.

## Gate Function: 5단계 검증

### 1. IDENTIFY — 검증 대상 식별
- 내가 "완료"라고 주장하려는 것이 정확히 무엇인가?
- 어떤 증거가 있어야 완료를 증명할 수 있는가?

### 2. RUN — 검증 실행
- 검증 명령어/프로세스를 **지금 이 시점에** 직접 실행한다
- 이전 실행 결과를 재사용하지 않는다

### 3. READ — 결과 확인
- 실행 결과를 **끝까지** 읽는다
- 성공 메시지만 보지 않고, 경고와 부분 실패도 확인한다

### 4. VERIFY — 요구사항 대조
- 결과가 원래 요구사항을 **모두** 충족하는지 확인한다
- 부분 충족은 완료가 아니다

### 5. CLAIM — 증거와 함께 완료 선언
- "완료했습니다" + 증거 = 유효한 완료
- "완료했습니다" (증거 없음) = 무효

## Definition of Done 체크리스트

### 공통 (모든 구현 작업)
| # | 항목 | 설명 |
|---|------|------|
| 1 | 에러 핸들링 | 모든 async 함수에 try-catch. 의미 있는 에러 메시지 |
| 2 | 하드코딩 금지 | 매직 넘버/문자열 → 상수 또는 환경변수 |
| 3 | AC 전체 통과 | Happy Path + 에러 케이스 + 엣지 케이스 |
| 4 | 빌드 성공 | `tsc --noEmit` + `pnpm lint` 에러 없음 |
| 5 | 런타임 테스트 | 실제 실행 확인 (타입 체크만으로 불충분) |

## QA 검증 워크플로우 (코드 변경 시 필수)
- **Backend**: Homer (화이트박스) → Chalmers (블랙박스 QA)
- **Frontend**: Bart (화이트박스) → Chalmers (블랙박스 QA)
- 런타임 테스트는 QA 단계의 책임

## 작업 유형별 검증

| 작업 유형 | 필수 검증 |
|-----------|----------|
| 코드 구현 | 테스트 + 빌드 + lint 통과 |
| API 개발 | 엔드포인트 테스트 + 스키마 검증 + 에러 케이스 |
| 코드 분석/보고 | 수치 → 소스 라인 대조 + 조건 분기 전수 분석 + 제안 시 원본 diff 비교 |
| 디자인 시안 | 스펙 체크리스트 + 접근성 + 반응형 |
| 보안 리뷰 | 취약점 스캔 + 재현 + 수정 확인 |
| 시장 조사 | 출처 교차 검증 + 데이터 유효성 + 최신성 |
| 기획 문서 | 요구사항 체크리스트 + 이해관계자 리뷰 + 기술 검토 |
| 버그 수정 | 원인 확인 + 수정 테스트 + 회귀 테스트 |

### 코드 분석/보고 검증 상세

수치 주장, 조건 분기 전수 분석, 최적화 제안 diff 비교 → `references/code-analysis-verification.md`

## Red Flags

| 핑계 | 현실 |
|------|------|
| "이제 작동할 것이다" | 검증 명령 실행해라 |
| "확신한다" | 확신 ≠ 증거 |
| "린터/타입체크 통과했다" | 린터 ≠ 런타임 |
| "에이전트가 성공이라 했다" | 독립적으로 검증해라 |
| "이전에 확인했다" | 이전 결과 ≠ 현재 증거 |
| "부분 검사로 충분하다" | 부분은 아무것도 증명 못함 |
