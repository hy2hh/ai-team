## 1. 버그별 상세 분석

### BUG-3: MCP slack_post_message 직접 호출 → 채널 본문 2중 포스팅 (HIGH)

**현상**: PM/QA 응답이 스레드가 아닌 채널 본문에 별도 메시지로 게시. 에이전트가 MCP 도구로 직접 Slack에 쓰고, bridge도 동일 응답을 포스팅하여 2중 게시 발생.

**근본 원인**: `permissionMode: 'bypassPermissions'` 환경에서 `allowedTools` 필터가 MCP 쓰기 도구를 차단하지 못함. `allowedTools`는 bypassPermissions 모드에서 우회되는 구조적 한계.

**현재 수정 상태**: 커밋 `cbdf78d4`에서 `disallowedTools`로 차단 완료. `MCP_SLACK_WRITE_TOOLS` 배열(`agent-runtime.ts:843-847`)을 정의하고, 비회의 모드에서 명시적 차단 적용(`agent-runtime.ts:2379`). `disallowedTools`는 모델 컨텍스트에서 아예 제거되므로 bypassPermissions 영향을 받지 않음.

**잔여 리스크**: 회의 모드(`isMeetingContext`)에서는 여전히 MCP 쓰기 도구 허용 — 회의 모드에서의 2중 포스팅 가능성 검증 필요.

**담당 에이전트**: Homer (Backend) — bridge 코드 소유자
**검증 에이전트**: Chalmers (QA) — E2E 재테스트로 2중 포스팅 미발생 확인

### BUG-5: Lisa 비-리서치 질문에 리서치 모드 진입 (MEDIUM)

**현상**: "@Lisa bridge 코드 분석해줘" 같은 코드 분석 요청에 리서치 A/B 모드 선택 대기 진입. 코드 분석은 즉시 답변 대상이므로 리서치 모드 불필요.

**근본 원인**: `isResearchRequest()` 함수(`agent-runtime.ts:454-473`)의 키워드 매칭 로직 문제. `researchKeywords`에 '분석'이 포함되어 있어, "코드 분석해줘"가 `nonResearchPatterns`의 '분석해줘'에 먼저 매칭되어야 하지만, "bridge 코드 분석해줘"처럼 중간에 다른 단어가 들어가면 '분석해줘' 패턴이 매칭되지 않고 '분석' 키워드에 매칭됨.

**현재 수정 상태**: 커밋 `b488a7c1`에서 `nonResearchPatterns`에 패턴 추가됨. 그러나 현재 코드 기준으로 "bridge 코드 분석해줘"는 '코드 분석' 패턴에 매칭되지 않을 수 있음 (텍스트: "bridge 코드 분석해줘" → '코드 분석' 부분 문자열 존재하므로 매칭 가능).

**잔여 리스크**: 
- "프론트엔드 분석해줘", "아키텍처 분석해줘" 등 코드 관련이지만 nonResearchPatterns에 없는 패턴
- 근본적으로 키워드 기반 판별의 한계 — LLM 기반 분류로 전환 검토 필요

**담당 에이전트**: Homer (Backend) — `isResearchRequest()` 로직 수정
**보조**: Lisa (Researcher) — 리서치 vs 비-리서치 경계 케이스 제공
**검증 에이전트**: Chalmers (QA) — 다양한 패턴으로 E2E 재테스트

---

### BUG-4: SESSION_TTL 체크리스트 72h vs 코드 30일 (LOW)

**현상**: E2E 체크리스트에는 SESSION_TTL이 72시간으로 기재되어 있으나, 실제 코드(`config.ts:22`)는 30일(`30 * 24 * 60 * 60 * 1000`)로 설정됨.

**근본 원인**: 스펙 문서와 구현 코드의 동기화 실패. 초기 설계 시 72시간으로 계획했으나, 구현 과정에서 30일로 변경하면서 문서 갱신 누락.

**현재 수정 상태**: 커밋 `b488a7c1`에서 체크리스트를 30일로 갱신 완료.

**잔여 리스크**: 없음 (문서 정합성 문제, 이미 수정됨).

**담당 에이전트**: Marge (PM) — 스펙 문서 관리자로서 정합성 최종 확인
**검증**: 별도 E2E 불필요 — 문서 diff 확인으로 충분
