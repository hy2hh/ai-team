# Systematic Debugging

> **Iron Law**: 수정 시도 전에 반드시 근본 원인을 찾아라. 증상 수정은 실패다.

## Phase 1: Root Cause Investigation

### 1.1 에러 메시지 완전 읽기
- 스택 트레이스를 **끝까지** 읽는다
- 에러 메시지의 모든 정보를 활용한다 (파일명, 라인 번호, 변수값)
- "대충 아는 것 같다"로 넘어가지 않는다

### 1.2 재현 확인
- 문제를 **일관되게** 재현할 수 있는가?
- 재현 조건을 정확히 기록한다 (입력값, 환경, 순서)
- 간헐적 문제 → 재현 조건을 좁히는 것이 최우선

### 1.3 최근 변경 확인
- `git diff`, `git log`로 최근 변경사항 확인
- "언제부터 문제가 발생했는가?"를 파악
- 변경 없이 갑자기 발생 → 외부 의존성, 환경 변화 확인

### 1.4 데이터 흐름 추적 (Root Cause Tracing)
```
증상 관찰 → 직접 원인 찾기 → "이것을 호출한 것은?" → 최초 트리거 발견 → 여기를 수정
```

### 1.5 팀 협업
- 다른 에이전트 작업이 원인일 수 있으면 `.memory/`에서 확인 → Slack @mention → 공동 디버깅

## Phase 2: Pattern Analysis

- 동작하는 유사 코드 찾기 → 정상 vs 비정상 차이점 목록화
- 공식 문서, 예제 코드와 비교 (부분만 보지 않는다)
- 환경 차이, 설정 차이, 의존성 버전 차이 확인
- 숨겨진 의존성 (전역 상태, 환경 변수, 실행 순서) 확인

## Phase 3: Hypothesis Testing

- 한 번에 **하나의** 가설만: "X 때문에 Y가 발생한다"
- **한 번에 하나의 변수만** 변경하여 테스트
- 가설이 틀렸다면 → 새로운 증거 기반 다음 가설
- 추측으로 코드를 수정하지 않는다

## Phase 4: Implementation

1. **실패 테스트 먼저** — 수정 전에 버그를 재현하는 테스트 작성
2. **단일 수정** — 근본 원인에 대한 단일 수정 구현
3. **검증** — 실패 테스트 통과 + 기존 테스트 회귀 없음
4. **방어적 계층** — 입력 검증, 비즈니스 로직 검증, 환경 가드, 디버그 계측

### 3회 규칙 (3-Strike Rule)
- 수정 시도 횟수를 추적한다
- **3회 실패 시**: 같은 접근 반복 금지
  1. 시도와 결과 정리
  2. `escalate_to_pm` 도구로 PM에게 에스컬레이션 (PM이 sid 판단 요청)
  3. "접근 자체가 틀렸다"를 의심

## Red Flags — Phase 1로 돌아가라

| 핑계 | 현실 |
|------|------|
| "일단 빠르게 고치자" | Phase 1부터 해라 |
| "이해 못하지만 될 수도" | 이해 없는 수정 = 새 버그 |
| "한 번 더 시도" (2+ 실패) | 접근 자체에 의문 제기 |
| "간단한 문제니까 바로 수정" | 간단해 보이는 곳에서 가장 큰 오류 |
| "여러 개 동시에 바꾸면 빠르다" | 한 번에 하나, 검증 후 다음 |

## 에스컬레이션

| 상황 | 행동 |
|------|------|
| 3+ 수정 실패 | `escalate_to_pm` → PM이 sid 에스컬레이션 판단 |
| 다른 에이전트 작업 원인 | 해당 에이전트 @mention + 공동 디버깅 |
| 재현 불가 | 디버그 계측 추가 + 모니터링 강화 |
| 외부 의존성 원인 | sid 보고 + 우회 방안 제시 |
