# Test-Driven Development

> **Iron Law**: 실패하는 테스트 없이 프로덕션 코드를 작성하지 않는다. 예외 없음.

## Red-Green-Refactor

### RED — 실패 테스트 작성
- 하나의 동작, 하나의 테스트
- 명확한 이름 (동작을 설명)
- 실제 코드 사용 (mock은 불가피할 때만)

### Verify RED — 실패 확인 (필수, 건너뛰기 금지)
```bash
pnpm test path/to/test.test.ts
```
확인 사항:
- 테스트가 **실패**하는가 (에러가 아닌 실패)
- 실패 메시지가 예상한 것인가
- 기능 미구현 때문에 실패하는가 (오타 아님)

**테스트가 통과하면?** → 기존 동작을 테스트하고 있음. 테스트 수정.

### GREEN — 최소 구현
- 테스트를 통과시키는 **가장 단순한** 코드
- 추가 기능, 리팩토링, 개선 금지
- YAGNI 준수

### Verify GREEN — 통과 확인 (필수)
```bash
pnpm test path/to/test.test.ts
```
확인 사항:
- 테스트 통과
- 다른 테스트 깨지지 않음
- 경고/에러 출력 없음

**테스트 실패 시?** → 코드 수정. 테스트 수정 아님.

### REFACTOR — 정리
- GREEN 이후에만
- 중복 제거, 이름 개선, 헬퍼 추출
- 테스트는 계속 GREEN 유지
- 새 동작 추가 금지

## Bug Fix 패턴
```
버그 발견 → 실패 테스트(버그 재현) → 수정 → 통과 → 리팩토링
```
테스트 없는 버그 수정 금지. `/agent-debug` Phase 4와 연계.

## Common Rationalizations

| 핑계 | 현실 |
|------|------|
| "너무 간단해서 테스트 불필요" | 간단한 코드도 깨진다. 테스트는 30초 |
| "나중에 테스트 추가" | 나중에 쓴 테스트는 즉시 통과 → 아무것도 증명 못함 |
| "이미 수동으로 확인했다" | 수동 = 비체계적. 기록 없고 재실행 불가 |
| "삭제하면 낭비" | 매몰 비용 오류. 신뢰 못하는 코드가 진짜 낭비 |
| "참고용으로 보존" | 그걸 적응시키는 순간 test-after. 삭제하고 TDD로 재작성 |
| "TDD는 교조적" | TDD가 실용적. 디버깅보다 빠름 |
| "먼저 탐색해야" | 탐색 후 버려라. TDD로 새로 시작 |
| "테스트가 어려움 = 스킵" | 테스트가 어려움 = 설계가 나쁨. 인터페이스 단순화 |
| "기존 코드에 테스트 없다" | 개선 중이다. 기존 코드에 테스트 추가 |

## Red Flags — 즉시 중단하고 TDD 재시작

- 테스트 전에 코드 작성
- 구현 후 테스트 추가
- 테스트가 즉시 통과
- 왜 실패했는지 설명 못함
- "이번만" 합리화
- "나중에 추가" 또는 "참고용으로 보존"
- "test-after도 같은 효과"

## Verification Checklist

- [ ] 모든 새 함수/메서드에 테스트 존재
- [ ] 각 테스트가 구현 전에 실패하는 것을 확인
- [ ] 각 실패가 예상한 이유인지 확인 (기능 미구현, 오타 아님)
- [ ] 최소 코드로 통과
- [ ] 전체 테스트 통과
- [ ] 출력 깨끗 (에러/경고 없음)
- [ ] 실제 코드 테스트 (mock이 아닌)
- [ ] 엣지 케이스와 에러 케이스 커버
