# Spec Compliance Review 가이드

## 핵심 원칙
**구현자의 보고를 믿지 마라. 코드를 직접 읽어라.**

구현자는 빠르게 완료했을 수 있다. 보고가 불완전하거나 낙관적일 수 있다.

## DO NOT (금지)
- 구현자의 "완료" 주장을 액면 그대로 수용
- 구현자의 완료도 해석을 그대로 신뢰
- 보고서만 읽고 리뷰 종료

## DO (필수)
- 실제 코드를 직접 읽기 (Read/Grep 도구 사용)
- 요구사항을 라인별로 대조
- 구현자가 주장한 내용이 실제로 코드에 있는지 확인
- 요구사항에 없는 추가 기능이 있는지 확인

## 검증 항목

### Missing Requirements (누락)
- 요구사항 중 구현되지 않은 것이 있는가?
- 구현자가 건너뛰거나 놓친 항목은?
- "구현했다"고 주장했지만 실제 코드에 없는 것은?

### Extra Work (불필요한 추가)
- 요청하지 않은 기능을 만들었는가?
- 과도하게 엔지니어링했는가?
- "nice to have"를 spec 없이 추가했는가?

### Misunderstandings (오해)
- 요구사항을 다르게 해석했는가?
- 잘못된 문제를 풀었는가?
- 맞는 기능을 틀린 방법으로 구현했는가?

## 보고 형식

```
✅ Spec compliant — 코드 검사 후 모든 요구사항 충족 확인
```
또는
```
❌ Issues found:
- 누락: [구체적 요구사항] (해당 코드 없음)
- 추가: [불필요한 기능] (file.ts:42)
- 오해: [잘못된 해석] (file.ts:78 — spec은 X인데 Y로 구현)
```

**반드시 file:line 참조 포함.**
