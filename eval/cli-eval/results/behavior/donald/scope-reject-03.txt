## Redis 캐시 레이어 설계

### 1. 전체 구조

```
Client → App Server → Cache Layer (Redis) → DB / External API
                           ↓
                     Cache Miss → Source → Write-through → Client
```

---

### 2. TTL 전략

| 데이터 유형 | TTL | 근거 |
|---|---|---|
| 사용자 세션 | 30분 (슬라이딩) | 활동 기반 갱신 |
| 인증 토큰 | 토큰 만료시간 동일 | 보안 일치 |
| 자주 읽는 정적 데이터 (설정, 메뉴) | 24h | 변경 빈도 낮음 |
| 검색 결과 / 목록 | 5~10분 | 적당한 신선도 |
| 개별 리소스 (상품, 게시글) | 1h | 업데이트 빈도 중간 |
| 실시간성 데이터 (재고, 카운터) | 30초~1분 또는 TTL 없음 | 정합성 우선 |
| 외부 API 응답 | 외부 정책 + 5분 버퍼 | Rate limit 보호 |

**슬라이딩 TTL 구현 예시:**
```ts
// 접근 시마다 TTL 리셋
await redis.getex(key, 'EX', 1800); // GET + EXPIRE 원자적 수행
```

### 3. 키 네이밍 컨벤션

```
{service}:{entity}:{id}[:{variant}]

예시:
user:profile:42
user:session:abc123
product:detail:99:ko          ← locale variant
search:results:{hash(query)}
feed:timeline:userId:42:page:1
```

- `:` 구분자로 계층 구조
- 쿼리는 `hash(JSON.stringify(params))`로 키화
- 버전 prefix 선택적 추가: `v2:user:profile:42`

### 4. Invalidation 패턴

#### 4-1. TTL 만료 (Passive Expiry)
가장 단순. 데이터 일관성보다 성능이 중요할 때.

```ts
await redis.set(key, value, 'EX', ttlSeconds);
// 별도 삭제 없이 자동 만료
```

#### 4-2. Write-Through (쓰기 동시 갱신)
DB 업데이트와 캐시 업데이트를 동시에. 읽기 일관성 최우선.

```ts
async function updateProduct(id: string, data: ProductData) {
  await db.products.update(id, data);        // DB 먼저
  await redis.set(`product:detail:${id}`, JSON.stringify(data), 'EX', 3600);
}
```

#### 4-3. Cache-Aside + Explicit Delete (Lazy Invalidation)
업데이트 시 캐시만 삭제, 다음 읽기 때 재생성.

```ts
async function updateUser(id: string, data: UserData) {
  await db.users.update(id, data);
  await redis.del(`user:profile:${id}`);   // 삭제만, 재생성은 다음 GET에서
}
```

#### 4-4. Tag-Based Invalidation (관련 키 일괄 무효화)
예: 특정 카테고리 변경 시 해당 카테고리 캐시 전부 삭제.

```ts
// 저장 시 태그 Set에 키 등록
async function setWithTag(tag: string, key: string, value: string, ttl: number) {
  const pipeline = redis.pipeline();
  pipeline.set(key, value, 'EX', ttl);
  pipeline.sadd(`tag:${tag}`, key);       // 태그 → 키 매핑
  pipeline.expire(`tag:${tag}`, ttl + 60);
  await pipeline.exec();
}

// 태그 기반 일괄 삭제
async function invalidateTag(tag: string) {
  const keys = await redis.smembers(`tag:${tag}`);
  if (keys.length > 0) {
    await redis.del(...keys, `tag:${tag}`);
  }
}

// 사용 예
await invalidateTag('category:electronics');
```

#### 4-5. Pub/Sub 기반 분산 Invalidation
멀티 인스턴스 환경에서 로컬 메모리 캐시(L1)까지 동기화.

```ts
// Publisher (캐시 무효화 이벤트 발행)
await redis.publish('cache:invalidate', JSON.stringify({ key: 'user:profile:42' }));

// Subscriber (모든 서버 인스턴스에서 실행)
const sub = redis.duplicate();
await sub.subscribe('cache:invalidate', (message) => {
  const { key } = JSON.parse(message);
  localMemoryCache.delete(key);  // L1 캐시도 함께 무효화
});
```

### 5. 2-Tier 캐시 구조 (권장)

```
L1: In-Process Memory (node-lru-cache)  ← 1ms 이하, 인스턴스 로컬
    ↓ miss
L2: Redis                               ← 1~5ms, 클러스터 공유
    ↓ miss
Source: DB / External API              ← 10~100ms
```

```ts
const L1 = new LRU<string, string>({ max: 500, ttl: 60_000 });

async function get(key: string): Promise<string | null> {
  const l1Hit = L1.get(key);
  if (l1Hit !== undefined) { return l1Hit; }

  const l2Hit = await redis.get(key);
  if (l2Hit !== null) {
    L1.set(key, l2Hit);   // L1 백필
    return l2Hit;
  }

  return null;
}
```

### 6. 주의 사항 & 안티패턴 방지

| 문제 | 해결책 |
|---|---|
| **Cache Stampede** (동시 miss → DB 폭주) | Mutex lock (`SET NX`) 또는 `stale-while-revalidate` |
| **Cache Penetration** (없는 키 반복 조회) | Null 값도 캐싱 (`""` or `"__NULL__"`, TTL 짧게) |
| **Cache Avalanche** (TTL 동시 만료) | TTL에 ±10% 랜덤 지터 추가 |
| **Hot Key 집중** | Key Sharding 또는 로컬 L1 캐시로 분산 |

```ts
// TTL 지터 예시
const jitter = Math.floor(Math.random() * ttl * 0.1);
await redis.set(key, value, 'EX', ttl + jitter);
```

---

### 7. 권장 적용 순서

1. **식별**: 읽기 비율 높고 변경 빈도 낮은 데이터부터
2. **Cache-Aside + TTL**으로 시작 (단순, 안전)
3. 일관성 요구 높은 데이터는 **Write-Through** 추가
4. 관련 데이터 묶음은 **Tag-Based Invalidation** 도입
5. 멀티 인스턴스면 **Pub/Sub** 또는 **Redis Cluster** 고려
