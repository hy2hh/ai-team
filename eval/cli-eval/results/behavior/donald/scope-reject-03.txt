# Redis 캐시 레이어 설계

## 전체 아키텍처

```
Client
  │
  ▼
Application Layer
  │
  ├─► Cache Layer (Redis)  ─── HIT ──► return cached value
  │         │
  │        MISS
  │         │
  └─► Data Source (DB / API)
            │
            └─► write-through / write-back ──► Redis
```

---

## 1. TTL 전략

데이터 특성에 따라 TTL 계층을 분리합니다.

| 계층 | 대상 | TTL | 이유 |
|------|------|-----|------|
| **Static** | 공통 코드, 설정값 | 24h | 변경 빈도 매우 낮음 |
| **Semi-static** | 사용자 프로필, 상품 정보 | 1h | 가끔 변경 |
| **Dynamic** | 세션, 장바구니 | 30m | 자주 변경 |
| **Real-time** | 재고, 가격 | 60s | 정합성 중요 |
| **Ephemeral** | Rate limit, OTP | 5m | 단기 사용 후 폐기 |

### TTL 상수 정의 (TypeScript 예시)

```typescript
/** Redis TTL 상수 (초 단위) */
export const CACHE_TTL = {
  STATIC: 86_400,       // 24h
  SEMI_STATIC: 3_600,   // 1h
  DYNAMIC: 1_800,       // 30m
  REAL_TIME: 60,        // 1m
  EPHEMERAL: 300,       // 5m
} as const;
```

### TTL Jitter (캐시 스탬피드 방지)

동일 TTL로 다수 키가 동시 만료되면 DB 스파이크 발생 → **±10% 랜덤 편차** 적용.

```typescript
/**
 * TTL에 랜덤 지터를 추가해 동시 만료를 방지합니다.
 * @param baseTtl - 기본 TTL (초)
 * @param jitterRatio - 지터 비율 (기본 10%)
 */
const withJitter = (baseTtl: number, jitterRatio = 0.1): number => {
  const delta = Math.floor(baseTtl * jitterRatio * Math.random());
  return baseTtl + delta;
};

// 사용
await redis.setex(key, withJitter(CACHE_TTL.SEMI_STATIC), value);
```

## 2. 캐시 키 네이밍 전략

```
{namespace}:{version}:{entity}:{id}:{variant?}

예시:
  user:v1:profile:1234
  product:v2:detail:5678
  product:v2:list:category=shoes:page=1
  session:v1:auth:abc-token
```

| 구성요소 | 목적 |
|----------|------|
| `namespace` | 서비스 또는 도메인 구분 |
| `version` | 스키마 변경 시 전체 무효화 가능 |
| `entity` | 데이터 종류 |
| `id` | 고유 식별자 |
| `variant` | 필터, 페이지 등 부가 파라미터 |

## 3. Invalidation 패턴

### 패턴 1. TTL 만료 (Passive Expiry)
- 별도 처리 없이 TTL 도달 시 자연 삭제
- **적합**: 정합성이 느슨해도 되는 데이터

### 패턴 2. Write-Through
쓰기 시 DB와 캐시를 동시에 업데이트.

```typescript
const updateUserProfile = async (userId: string, data: UserProfile) => {
  // 1. DB 업데이트
  await db.user.update({ where: { id: userId }, data });

  // 2. 캐시 즉시 갱신
  const key = `user:v1:profile:${userId}`;
  await redis.setex(key, withJitter(CACHE_TTL.SEMI_STATIC), JSON.stringify(data));
};
```
- **장점**: 캐시 항상 최신 / **단점**: 쓰기 지연 증가

### 패턴 3. Cache-Aside (Lazy Loading)
읽기 시 캐시 미스이면 DB 조회 후 캐시 저장.

```typescript
/**
 * 사용자 프로필을 캐시 우선으로 조회합니다.
 */
const getUserProfile = async (userId: string): Promise<UserProfile> => {
  const key = `user:v1:profile:${userId}`;

  const cached = await redis.get(key);
  if (cached) {
    return JSON.parse(cached);
  }

  const data = await db.user.findUnique({ where: { id: userId } });
  await redis.setex(key, withJitter(CACHE_TTL.SEMI_STATIC), JSON.stringify(data));

  return data;
};
```

### 패턴 4. Event-Driven Invalidation (권장)
이벤트 발행 시 관련 키를 능동적으로 삭제.

```typescript
/** 사용자 관련 캐시 키 패턴 */
const getUserCacheKeys = (userId: string) => [
  `user:v1:profile:${userId}`,
  `user:v1:settings:${userId}`,
  `user:v1:permissions:${userId}`,
];

/**
 * 사용자 데이터 변경 이벤트 핸들러 — 관련 캐시를 무효화합니다.
 */
const onUserUpdated = async (userId: string) => {
  const keys = getUserCacheKeys(userId);
  await redis.del(...keys);
};
```

### 패턴 5. Tag-Based Invalidation
연관 키를 태그로 묶어 일괄 무효화.

```typescript
/**
 * 태그에 연결된 캐시 키를 등록합니다.
 */
const tagCache = async (tag: string, key: string, ttl: number) => {
  await redis.sadd(`tag:${tag}`, key);
  await redis.expire(`tag:${tag}`, ttl);
};

/**
 * 태그에 속한 모든 캐시 키를 무효화합니다.
 */
const invalidateByTag = async (tag: string) => {
  const keys = await redis.smembers(`tag:${tag}`);
  if (keys.length > 0) {
    await redis.del(...keys, `tag:${tag}`);
  }
};

// 예: 상품 목록 캐시 등록
await tagCache('product:category:shoes', 'product:v2:list:category=shoes:page=1', CACHE_TTL.SEMI_STATIC);

// 예: 상품 변경 시 카테고리 전체 무효화
await invalidateByTag('product:category:shoes');
```

## 4. 캐시 스탬피드 방지 — Probabilistic Early Recomputation

TTL 만료 직전에 확률적으로 미리 갱신해 동시 만료 방지.

```typescript
/**
 * 확률적 조기 갱신으로 캐시 스탬피드를 방지합니다.
 * @param key - 캐시 키
 * @param fetcher - 실제 데이터 조회 함수
 * @param ttl - TTL (초)
 * @param beta - 조기 갱신 확률 조정값 (기본 1.0)
 */
const getWithEarlyRecompute = async <T>(
  key: string,
  fetcher: () => Promise<T>,
  ttl: number,
  beta = 1.0,
): Promise<T> => {
  const raw = await redis.get(key);

  if (raw) {
    const { value, expireAt } = JSON.parse(raw) as { value: T; expireAt: number };
    const now = Date.now() / 1000;
    const remaining = expireAt - now;

    // 남은 TTL이 적을수록 재계산 확률 증가
    if (remaining > beta * Math.log(Math.random()) * -1) {
      return value;
    }
  }

  const value = await fetcher();
  const expireAt = Date.now() / 1000 + ttl;
  await redis.setex(key, ttl, JSON.stringify({ value, expireAt }));

  return value;
};
```

## 5. 운영 가이드라인

| 항목 | 권장 설정 |
|------|-----------|
| **메모리 정책** | `allkeys-lru` (메모리 부족 시 LRU 제거) |
| **Persistence** | AOF `appendfsync everysec` (성능·안정성 균형) |
| **Key 분리** | 환경별 prefix (`prod:`, `dev:`, `test:`) |
| **모니터링** | `redis-cli monitor`, `INFO stats`로 HIT/MISS율 추적 |
| **Circuit Breaker** | Redis 장애 시 fallback → DB 직접 조회 |

## 요약

```
읽기 흐름:  Cache-Aside (Lazy Loading) + Early Recomputation
쓰기 흐름:  Write-Through (정합성 중요) or TTL 만료 (느슨한 경우)
무효화:     Event-Driven Delete + Tag-Based 일괄 무효화
TTL:        계층별 차등 + Jitter로 스탬피드 방지
키 설계:    namespace:version:entity:id 형식으로 일관성 유지
```
