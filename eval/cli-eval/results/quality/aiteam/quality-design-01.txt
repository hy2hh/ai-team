# 🔐 사용자 인증 시스템 — Feature Spec

**작성자**: Marge (PM) | **날짜**: 2026-04-09 | **상태**: Draft

---

## 1. 문제 정의

> 인증은 제품의 신뢰 기반이다. 잘못 설계된 인증은 사용자 이탈, 보안 사고, 법적 리스크를 동시에 유발한다.

**핵심 사용자 페인포인트:**
- 이메일/비밀번호만 지원 시 → "또 계정 만들어야 해?" 마찰로 이탈
- 비밀번호 분실 처리가 느리거나 불안하면 → 신뢰 손상
- OAuth 미지원 시 → 개발자/기술 유저 타겟에서 치명적 (GitHub)

**비즈니스 목표:**
- 회원가입 완료율 80% 이상 달성
- 로그인 관련 CS 문의 월 50건 이하 유지
- 보안 사고 0건 (OWASP Top 10 기준)

## 3. 사용자 스토리

### 3-1. 이메일/비밀번호 인증

```
As a 신규 사용자,
I want to 이메일과 비밀번호로 계정을 만들고,
So that 서비스를 이용할 수 있다.

인수 조건:
- [ ] 이메일 형식 유효성 검사 (실시간)
- [ ] 비밀번호 강도 표시기 (weak / fair / strong)
- [ ] 비밀번호 최소 조건: 8자 이상, 대소문자+숫자+특수문자 중 3종 조합
- [ ] 가입 완료 후 이메일 인증 링크 발송 (24시간 유효)
- [ ] 미인증 상태에서 로그인 시 재발송 옵션 제공
- [ ] 동일 이메일 중복 가입 시 명확한 에러 메시지 ("이미 사용 중인 이메일입니다. 로그인하거나 비밀번호를 재설정하세요.")
```

```
As a 기존 사용자,
I want to 이메일과 비밀번호로 로그인하고,
So that 내 데이터에 접근할 수 있다.

인수 조건:
- [ ] 로그인 실패 5회 이상 → 15분 잠금 + 이메일 알림
- [ ] "로그인 유지" 체크박스 (30일 세션 vs 브라우저 세션)
- [ ] 잘못된 자격증명 시 이메일/비밀번호 구분 없는 에러 ("이메일 또는 비밀번호가 올바르지 않습니다" — 계정 존재 여부 노출 금지)
```

### 3-2. OAuth 인증

```
As a 사용자,
I want to Google 또는 GitHub 계정으로 로그인하고,
So that 별도 계정 생성 없이 빠르게 시작할 수 있다.

인수 조건:
- [ ] Google OAuth 2.0 / GitHub OAuth 앱 연동
- [ ] OAuth 최초 로그인 시 자동 계정 생성 (이메일 인증 불필요)
- [ ] 동일 이메일로 이미 이메일/비밀번호 계정 존재 시:
      → "이미 이메일로 가입된 계정이 있습니다. 로그인 후 Google 연결을 추가할 수 있습니다." 안내
- [ ] OAuth 실패(권한 거부 등) 시 홈으로 복귀 + 안내 메시지
- [ ] 반환받은 프로필(이름, 이메일, 아바타) 기본값 저장
```

### 3-3. 비밀번호 재설정

```
As a 비밀번호를 잊은 사용자,
I want to 이메일로 재설정 링크를 받고,
So that 계정을 복구할 수 있다.

인수 조건:
- [ ] 가입되지 않은 이메일 입력 시에도 동일한 "발송했습니다" 응답 (계정 존재 여부 노출 금지)
- [ ] 재설정 링크: 1회용, 30분 유효
- [ ] 링크 만료 시 재발송 페이지로 안내
- [ ] 재설정 완료 후 기존 세션 전체 무효화
- [ ] 재설정 링크 발송은 10분당 최대 3회 (스팸/어뷰징 방지)
```

## 4. 보안 요구사항

### 4-1. 비밀번호 저장

| 요구사항 | 스펙 |
|---------|------|
| 해싱 알고리즘 | **bcrypt** (cost factor 12 이상) 또는 Argon2id |
| 평문 저장 | 절대 금지 — 로그, DB, 캐시 모두 |
| 비밀번호 힌트 | 제공 금지 |

### 4-2. 세션 & 토큰

| 항목 | 스펙 |
|------|------|
| Access Token | JWT, 만료 15분 |
| Refresh Token | Opaque token (DB 저장), 30일 / 회전(rotation) 방식 |
| 토큰 저장 위치 | Access: 메모리(in-memory) / Refresh: HttpOnly Secure Cookie |
| CSRF 방지 | SameSite=Strict + CSRF 토큰 (필요 시) |
| 토큰 재사용 감지 | Refresh Token 재사용 시 해당 사용자 전체 세션 무효화 |

### 4-3. 전송 보안

- **HTTPS 강제** — HTTP 요청은 301 리다이렉트
- **HSTS** 헤더 적용 (max-age=31536000)
- OAuth state 파라미터 검증 (CSRF 방지)
- 인증 관련 엔드포인트 Rate Limit:

| 엔드포인트 | 제한 |
|-----------|------|
| POST /auth/login | IP당 분당 10회 |
| POST /auth/register | IP당 시간당 5회 |
| POST /auth/password-reset | 이메일당 10분에 3회 |
| POST /auth/token/refresh | IP당 분당 30회 |

### 4-4. 감사 로그 (Audit Log)

다음 이벤트는 반드시 기록 (사용자 ID, IP, User-Agent, 타임스탬프):
- 로그인 성공/실패
- 비밀번호 변경/재설정
- OAuth 계정 연결
- 비정상 로그인 시도 감지
- 세션 강제 만료

## 5. API 계약 (개요)

```
POST /auth/register          # 이메일 가입
POST /auth/login             # 이메일 로그인
POST /auth/logout            # 로그아웃 (세션 무효화)
POST /auth/token/refresh     # Access Token 재발급
GET  /auth/verify-email      # 이메일 인증 링크 처리
POST /auth/password-reset    # 재설정 이메일 발송 요청
POST /auth/password-reset/confirm  # 새 비밀번호 설정

GET  /auth/oauth/google      # Google OAuth 시작
GET  /auth/oauth/google/callback
GET  /auth/oauth/github      # GitHub OAuth 시작
GET  /auth/oauth/github/callback
```

> 상세 Request/Response 스키마는 Backend(Homer) + Frontend(Bart) API Contract 협의 후 확정.

## 6. 완료 조건 (Definition of Done)

### 기능
- [ ] 이메일/비밀번호 가입 → 이메일 인증 → 로그인 플로우 E2E 동작
- [ ] Google / GitHub OAuth 로그인 동작
- [ ] 비밀번호 재설정 플로우 동작
- [ ] 로그인 유지 / 세션 만료 처리 동작

### 보안
- [ ] OWASP Top 10 항목 점검 완료 (Wiggum 리뷰)
- [ ] Rate limiting 동작 확인
- [ ] Refresh Token rotation 동작 확인
- [ ] 계정 존재 여부 노출 없음 확인

### 비기능
- [ ] 로그인 API P99 응답시간 < 500ms
- [ ] 이메일 발송 < 1분 이내 도달
- [ ] 접근성: 폼 레이블, 에러 메시지 스크린리더 동작

## 7. 구현 순서 추천

```
Phase 1 (필수, 먼저 출시)
  → 이메일/비밀번호 인증 + 이메일 인증 + 비밀번호 재설정

Phase 2 (빠른 후속)
  → Google OAuth + GitHub OAuth

Phase 3 (v2)
  → MFA (TOTP), 소셜 계정 연동 관리 UI
```
