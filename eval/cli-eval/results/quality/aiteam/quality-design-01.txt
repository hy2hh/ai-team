# 사용자 인증 시스템 설계

## 🎯 제품 목표

**핵심 문제**: 사용자가 안전하고 편리하게 서비스에 접근할 수 있어야 한다.
**성공 지표**: 인증 완료율 ≥ 95%, 비밀번호 재설정 완료율 ≥ 80%, 보안 사고 0건

---

## 📐 시스템 아키텍처

```
클라이언트 (Web/Mobile)
    │
    ▼
API Gateway (Rate Limiting + WAF)
    │
    ├── /auth/email     → 이메일 인증 서비스
    ├── /auth/oauth     → OAuth 서비스 (Google, GitHub)
    ├── /auth/reset     → 비밀번호 재설정 서비스
    └── /auth/token     → 토큰 관리 서비스
    │
    ▼
Auth Service
    │
    ├── User Store (DB)         → users, credentials
    ├── Session Store (Redis)   → refresh tokens, blacklist
    └── Email Service           → 재설정 메일, 인증 메일
```

## 🔐 1. 이메일/비밀번호 인증

### 회원가입 플로우

```
1. POST /auth/signup
   → 이메일 형식 검증
   → 중복 이메일 체크
   → 비밀번호 강도 검증
   → bcrypt(cost=12) 해시 저장
   → 이메일 인증 링크 발송 (토큰 TTL: 24시간)

2. GET /auth/verify-email?token={jwt}
   → 토큰 검증 (서명, 만료, 1회성)
   → 계정 활성화
```

### 로그인 플로우

```
1. POST /auth/login
   Body: { email, password }

2. 서버 처리:
   a. 이메일로 사용자 조회 (없으면 동일 응답 시간 유지 — timing attack 방지)
   b. bcrypt.compare(input, stored_hash)
   c. 실패 시: 실패 횟수 기록 (Redis)
   d. 5회 실패 → 15분 계정 잠금 (exponential backoff)
   e. 성공 시: Access Token (15분) + Refresh Token (7일) 발급

3. 응답:
   {
     access_token: "JWT (15분)",
     refresh_token: "opaque token (HttpOnly Cookie)",
     user: { id, email, name }
   }
```

### 비밀번호 정책

| 항목 | 기준 |
|------|------|
| 최소 길이 | 8자 이상 |
| 복잡도 | 대문자, 소문자, 숫자, 특수문자 중 3가지 이상 |
| 금지 패턴 | 이메일 포함, 연속 문자(aaa, 123), 상위 10만 취약 비밀번호 |
| 히스토리 | 최근 5개 비밀번호 재사용 금지 |
| 만료 | 선택적 (보안 요구사항에 따라) |

## 🌐 2. OAuth 인증 (Google, GitHub)

### OAuth 2.0 Authorization Code Flow

```
1. 사용자: "Google로 로그인" 클릭
2. 클라이언트 → POST /auth/oauth/init
   { provider: "google" }
   ← { redirect_url, state (CSRF token) }

3. 사용자 → Google 동의 화면
4. Google → GET /auth/oauth/callback?code=xxx&state=xxx
   → state 검증 (CSRF 방지)
   → code로 access_token 교환
   → 사용자 프로필 조회 (email, name, avatar)

5. 계정 연결 로직:
   ┌─────────────────────────────────────┐
   │ Google email로 기존 계정 있음?       │
   │   YES → OAuth credential 연결       │
   │   NO  → 신규 계정 생성              │
   └─────────────────────────────────────┘

6. 내부 JWT 발급 (이메일 로그인과 동일)
```

### 프로바이더별 스코프

```
Google:
  scope: openid email profile
  필요 데이터: sub, email, name, picture

GitHub:
  scope: user:email read:user
  필요 데이터: id, email, login, avatar_url
  주의: email이 private일 경우 /user/emails API 추가 호출
```

### 계정 연결 정책

- 동일 이메일 → 자동 연결 (기존 계정에 OAuth 추가)
- 이메일 충돌 시 사용자에게 연결 여부 명시적 확인
- 한 계정에 다중 OAuth 프로바이더 연결 허용
- OAuth 계정 연결 해제 가능 (단, 비밀번호 없는 계정은 불가)

## 🔄 3. 비밀번호 재설정

### 플로우

```
Step 1: 재설정 요청
POST /auth/forgot-password
{ email: "user@example.com" }
→ 응답: 항상 200 OK (이메일 존재 여부 노출 금지)
→ 이메일 존재 시: 재설정 링크 발송

Step 2: 토큰 구조
재설정 토큰 = HMAC-SHA256(userId + timestamp + secret)
TTL: 1시간
1회 사용 후 즉시 무효화 (Redis에서 삭제)

Step 3: 비밀번호 변경
POST /auth/reset-password
{ token, new_password }
→ 토큰 검증 (서명 + 만료 + 미사용)
→ 새 비밀번호 정책 검증
→ 해시 저장
→ 전체 세션 무효화 (다른 기기 로그아웃)
→ 완료 알림 이메일 발송

Step 4: 이메일 내용
제목: "[서비스명] 비밀번호 재설정 링크"
링크: https://app.example.com/reset?token=xxx
만료: 1시간
경고: 요청하지 않았다면 무시하세요
```

## 🔑 4. 토큰 관리

### Access Token (JWT)

```json
Header: { alg: "RS256", typ: "JWT" }
Payload: {
  sub: "user_id",
  email: "user@example.com",
  roles: ["user"],
  iat: 1712345678,
  exp: 1712346578,  // 15분
  jti: "unique_token_id"
}
```

### Refresh Token

- Opaque random token (256-bit)
- HttpOnly + Secure + SameSite=Strict 쿠키
- DB/Redis에 해시 저장 (토큰 자체는 저장 안 함)
- Rotation: 사용 시마다 새 Refresh Token 발급 (이전 것 무효화)
- Reuse Detection: 이미 사용된 토큰 재사용 감지 → 전체 세션 무효화

### 토큰 갱신

```
POST /auth/refresh
Cookie: refresh_token=xxx
→ 검증 → 새 Access Token + 새 Refresh Token
→ 이전 Refresh Token 무효화
```

## 🛡️ 5. 보안 요구사항

### API 보안

| 보안 항목 | 구현 |
|-----------|------|
| Rate Limiting | 로그인: IP당 10회/분, 재설정: 이메일당 3회/시간 |
| HTTPS | 전 구간 TLS 1.2+ 강제 |
| HSTS | max-age=31536000; includeSubDomains |
| CORS | 허용 도메인 화이트리스트 |
| Content-Security-Policy | strict-dynamic |

### 데이터 보안

```
비밀번호: bcrypt (cost factor 12) — 절대 평문 저장 금지
토큰 시크릿: AWS KMS / HashiCorp Vault 관리
개인정보: 암호화 at rest (AES-256)
로그: 비밀번호, 토큰 절대 로깅 금지
```

### 공격 방어

| 공격 유형 | 방어 전략 |
|-----------|-----------|
| Brute Force | 로그인 실패 횟수 제한 + 계정 잠금 |
| Credential Stuffing | IP 평판 + CAPTCHA (실패 3회 이상) |
| CSRF | SameSite Cookie + CSRF 토큰 |
| XSS | HttpOnly Cookie + CSP |
| Timing Attack | 상수 시간 비교 (crypto.timingSafeEqual) |
| Session Fixation | 로그인 성공 시 세션 ID 재생성 |
| Token Leakage | Refresh Token Rotation + Reuse Detection |

### 모니터링 & 감사 로그

```
필수 이벤트 기록:
- 로그인 성공/실패 (IP, User-Agent, timestamp)
- 비밀번호 변경 (이전 해시 보관 금지, 이벤트만)
- OAuth 연결/해제
- 계정 잠금/해제
- 비정상 접근 패턴 (새 기기, 새 국가)

알림 트리거:
- 동시 다국가 로그인
- 1분 내 5회 실패
- 비밀번호 재설정 후 즉시 다른 기기 로그인
```

## 📊 6. 데이터 모델

```sql
-- 사용자
CREATE TABLE users (
  id            UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  email         VARCHAR(255) UNIQUE NOT NULL,
  email_verified BOOLEAN DEFAULT false,
  created_at    TIMESTAMPTZ DEFAULT now(),
  locked_until  TIMESTAMPTZ,
  failed_attempts INT DEFAULT 0
);

-- 이메일/비밀번호 인증
CREATE TABLE credentials (
  user_id       UUID REFERENCES users(id),
  password_hash VARCHAR(60) NOT NULL,  -- bcrypt output
  updated_at    TIMESTAMPTZ DEFAULT now(),
  PRIMARY KEY (user_id)
);

-- OAuth 연결
CREATE TABLE oauth_accounts (
  id            UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  user_id       UUID REFERENCES users(id),
  provider      VARCHAR(20) NOT NULL,  -- 'google', 'github'
  provider_id   VARCHAR(255) NOT NULL,
  created_at    TIMESTAMPTZ DEFAULT now(),
  UNIQUE (provider, provider_id)
);

-- Refresh Token (해시만 저장)
CREATE TABLE refresh_tokens (
  id            UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  user_id       UUID REFERENCES users(id),
  token_hash    VARCHAR(64) NOT NULL,
  device_info   JSONB,
  expires_at    TIMESTAMPTZ NOT NULL,
  created_at    TIMESTAMPTZ DEFAULT now(),
  used_at       TIMESTAMPTZ  -- Reuse Detection용
);
```

**⚠️ 주요 설계 결정 & 트레이드오프**

> **JWT vs Session**: Access Token을 stateless JWT로 결정. 이점: 수평 확장 용이, MSA 친화적. 비용: 로그아웃 즉시 무효화 불가 → Access Token TTL을 15분으로 단축하여 완화.

> **Refresh Token 저장 위치**: HttpOnly Cookie (localStorage 대비 XSS 방어). CSRF는 SameSite=Strict로 방어.

> **이메일 인증 필수 여부**: MVP에서는 미인증 계정도 제한적 사용 허용을 권장. 즉시 강제 인증은 전환율을 최대 30% 낮출 수 있음.

구현 착수 전 Backend(Homer)와 아키텍처 확인, Wiggum(SecOps)의 보안 리뷰를 거치는 것을 권장합니다.
