### 🏗️ Homer (Backend) 관점

**DB 스키마 (최소)**
```sql
-- 고객 매핑
stripe_customers (
  id, user_id FK, stripe_customer_id UNIQUE, created_at
)

-- 구독 상태
stripe_subscriptions (
  id, user_id FK, stripe_subscription_id UNIQUE,
  stripe_price_id,
  status  -- trialing | active | past_due | canceled | unpaid
  current_period_end,
  cancel_at_period_end BOOLEAN,
  created_at, updated_at
)

-- 결제 이력 (Webhook으로 자동 기록)
stripe_invoices (
  id, user_id FK, stripe_invoice_id UNIQUE,
  amount_paid, currency, status, paid_at
)
```

**API 엔드포인트 (최소 4개)**
```
POST /api/stripe/checkout-session    → 결제 세션 생성
POST /api/stripe/portal-session      → 구독 관리 포털 세션 생성
GET  /api/subscriptions/me           → 내 구독 상태 조회
POST /api/webhooks/stripe            → Stripe → 서버 이벤트 수신
```

**Webhook 이벤트 (필수 처리 5개)**
```
checkout.session.completed         → subscription 활성화
customer.subscription.updated      → 플랜 변경/취소 예약
customer.subscription.deleted      → 구독 완전 해지
invoice.payment_succeeded          → 갱신 성공 기록
invoice.payment_failed             → 갱신 실패 → 사용자 알림 트리거
```

**Webhook Idempotency**: 처리된 이벤트 ID를 DB 저장해서 중복 처리 방지.

**Stripe Checkout 핵심 코드**
```typescript
// 서버: 세션 생성
const session = await stripe.checkout.sessions.create({
  mode: 'subscription',
  customer: stripeCustomerId,
  line_items: [{ price: priceId, quantity: 1 }],
  success_url: `${baseUrl}/dashboard?checkout=success`,
  cancel_url: `${baseUrl}/pricing`,
});

// 클라이언트: 리다이렉트
const stripe = await loadStripe(process.env.NEXT_PUBLIC_STRIPE_KEY);
await stripe.redirectToCheckout({ sessionId: session.id });
```

### 🔒 Wiggum (SecOps) 관점

**PCI DSS 범위**
- Stripe.js 사용 시 → **SAQ A 레벨** (가장 낮은 등급, 서버에 카드 데이터 없음)
- 카드번호/CVV/만료일은 절대 서버 통과 금지 → Stripe 토큰/PaymentMethod ID만 처리

**Webhook 보안 (필수)**
```typescript
// stripe-signature 헤더 검증 필수
const event = stripe.webhooks.constructEvent(
  req.rawBody,           // ← 반드시 raw body (JSON.parse 전)
  req.headers['stripe-signature'],
  process.env.STRIPE_WEBHOOK_SECRET
);
// 실패 시 400 반환, 처리 중단
```

**보안 체크리스트**
- [ ] `STRIPE_SECRET_KEY` 환경변수 전용 (절대 코드 하드코딩 금지)
- [ ] Webhook 엔드포인트 Rate Limiting 적용
- [ ] HTTPS 강제 (HTTP → HTTPS 리다이렉트)
- [ ] `stripe_customer_id` 클라이언트 노출 금지 (Portal URL은 서버에서 생성)
- [ ] 구독 상태 검증을 서버에서만 수행 (클라이언트 상태 신뢰 금지)

### 🔭 Lisa (Research) 관점

**Stripe vs 대안 비교**

| 솔루션 | 수수료 | 난이도 | 추천 시나리오 |
|--------|--------|--------|-------------|
| **Stripe Billing** | 0.5%/거래 | 중 | 스타트업 → 중견, 표준 선택 |
| Chargebee | $599/월~ | 낮음 | 대기업, 복잡한 플랜 구조 |
| Paddle | 5% + $0.5 | 낮음 | 글로벌 MoR 필요 시 |
| Lemon Squeezy | 5% | 매우 낮음 | 인디 제품, 간단한 구조 |

→ **Stripe Billing이 정답**. 수수료 최저, 문서 최고, 국내 연동 레퍼런스 풍부.

**SaaS 구독 UX 베스트 프랙티스**
1. **연간/월간 토글**: 연간 할인 20% 표시 (연간 ARR 증가에 효과적)
2. **Free Trial**: 7-14일 (카드 등록 요구 → 전환율 높음)
3. **취소 방지**: 취소 전 "1개월 일시정지" 옵션 제시
4. **결제 실패 복구**: 이메일 알림 → 앱 내 배너 → Grace Period 7일

## 🎯 접근 방식 3가지 제안 (sid 결정 필요)

### Option A: **Stripe Checkout + Customer Portal** ← *PM 추천 (MVP)*
- **개발 기간**: ~3-5일
- **방식**: 결제는 Stripe Checkout, 관리는 Customer Portal (모두 Stripe 호스팅 페이지)
- **장점**: 가장 빠른 출시, PCI 범위 최소화, edge case(3D Secure 등) 자동 처리
- **단점**: 브랜드 UI에서 벗어남, 커스터마이징 제한
- **추천 이유**: 검증 안 된 제품에 빠른 출시가 우선. 나중에 임베디드로 마이그레이션 쉬움.

### Option B: **Payment Element (완전 임베디드)**
- **개발 기간**: ~2-3주
- **방식**: 앱 내 결제 폼 + 구독 관리 화면 직접 구현
- **장점**: 완전한 브랜드 UX, 화면 이탈 없음
- **단점**: 구현 복잡도 3배↑, 결제 실패/재시도 로직 직접 구현 필요
- **추천 이유**: PMF 확인 후 성장 단계에서 고려

### Option C: **하이브리드 (Checkout → Portal → 이후 임베디드 전환)**
- **개발 기간**: ~1주
- **방식**: Option A로 시작, 단계적으로 임베디드 전환
- **장점**: 빠른 출시 + 성장에 따른 점진적 개선 가능
- **추천 이유**: 현실적인 중간 경로

## 📋 PM 추천 결정

**Option A (Stripe Checkout + Customer Portal)** 로 v1 출시 추천.

**근거**:
1. 아직 구독 UX가 검증되지 않은 상태에서 빠른 출시 > 완벽한 UI
2. Stripe Checkout은 이미 수억 명이 검증한 결제 UX
3. PCI 컴플라이언스 리스크 최소화 (Wiggum 관점)
4. 구현 시간 5일 vs 3주 → Opportunity Cost 명확

**다음 행동**: sid의 두 가지 확인 후 Feature Spec 작성 즉시 시작
