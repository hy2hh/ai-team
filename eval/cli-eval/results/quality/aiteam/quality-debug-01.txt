## 원인 분석

### 🔴 핵심 원인: Sequential Waterfall (직렬 쿼리)

```
현재 실행 순서:
users  ───────── 500ms ─────────►
                                  orders  ─── 500ms ───►
                                                         notifications ─── 500ms ───►
                                                                                       analytics ─── 500ms ───►
총 합산: ~2,000ms
```

4개의 독립적인 쿼리를 **순차적으로 `await`** 하고 있습니다. 각 쿼리가 완전히 끝날 때까지 다음 쿼리를 시작하지 않아 지연이 누적됩니다.

### 🟡 부차 원인들

| 문제 | 위치 | 영향 |
|------|------|------|
| `SELECT *` 남용 | 모든 쿼리 | 불필요한 컬럼 전송 오버헤드 |
| orders 무제한 조회 | orders 쿼리 | 수천 건 반환 가능 |
| 인덱스 부재 추정 | `user_id`, `read` 컬럼 | Full scan 가능성 |
| 캐싱 없음 | analytics | 동일 데이터 매 요청마다 재계산 |

## 해결책

### ✅ 1순위: `Promise.all()` 병렬화 — 즉각 ~75% 개선

```javascript
app.get('/api/dashboard', async (req, res) => {
  const [user, orders, notifications, analytics] = await Promise.all([
    db.query(
      'SELECT id, name, email, avatar_url FROM users WHERE id = ?',
      [req.userId]
    ),
    db.query(
      'SELECT id, status, total, created_at FROM orders WHERE user_id = ? ORDER BY created_at DESC LIMIT 10',
      [req.userId]
    ),
    db.query(
      'SELECT id, message, created_at FROM notifications WHERE user_id = ? AND read = false LIMIT 20',
      [req.userId]
    ),
    db.query(
      'SELECT COUNT(*) as total FROM page_views WHERE user_id = ?',
      [req.userId]
    ),
  ]);

  res.json({ user, orders, notifications, analytics });
});
```

```
병렬 실행 후:
users        ─── 500ms ───►
orders       ─── 500ms ───►  → 총 합산: ~500ms (가장 느린 쿼리 하나)
notifications─── 500ms ───►
analytics    ─── 500ms ───►
```

### ✅ 2순위: DB 인덱스 추가

```sql
-- orders.user_id 인덱스 (조회 빈도 높음)
CREATE INDEX idx_orders_user_id ON orders(user_id);

-- notifications 복합 인덱스 (user_id + read 필터 동시 적용)
CREATE INDEX idx_notifications_user_unread 
  ON notifications(user_id, read);

-- page_views COUNT 쿼리용
CREATE INDEX idx_page_views_user_id ON page_views(user_id);
```

> **체크 방법**: `EXPLAIN SELECT ...` 실행 후 `type: ALL` (Full Scan) 이면 인덱스 누락입니다.

### ✅ 3순위: analytics 캐싱 (조회 빈도 높을 때)

```javascript
const ANALYTICS_TTL = 60 * 1000; // 1분
const analyticsCache = new Map();

const getAnalytics = async (userId) => {
  const cached = analyticsCache.get(userId);
  if (cached && Date.now() - cached.ts < ANALYTICS_TTL) {
    return cached.data;
  }
  const data = await db.query(
    'SELECT COUNT(*) as total FROM page_views WHERE user_id = ?',
    [userId]
  );
  analyticsCache.set(userId, { data, ts: Date.now() });
  return data;
};
```

프로덕션 규모라면 Redis로 교체 권장.

## 예상 개선 효과

| 단계 | 조치 | 예상 응답 시간 |
|------|------|--------------|
| 현재 | 직렬 쿼리 | ~2,000ms |
| 1단계 후 | Promise.all 병렬화 | ~500ms |
| 2단계 후 | 인덱스 추가 | ~100–200ms |
| 3단계 후 | 캐싱 추가 | ~50–100ms |

**가장 먼저 `Promise.all()` 적용하세요.** 코드 한 줄 수정으로 즉각 4배 개선이 가능합니다.
