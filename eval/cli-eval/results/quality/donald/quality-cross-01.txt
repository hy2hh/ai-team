# Stripe 월간 구독 결제 시스템 — 전체 계획

---

## 1. 기획 (Product)

### 핵심 사용자 흐름
```
회원가입/로그인
  → 플랜 선택 (Free / Pro / Enterprise)
  → 결제 정보 입력 (Stripe Elements)
  → 구독 활성화 확인
  → 대시보드 (구독 상태, 청구 내역, 플랜 변경, 해지)
```

### 플랜 설계 (예시)
| 플랜 | 가격 | 핵심 기능 |
|------|------|-----------|
| Free | ₩0/월 | 기본 기능 |
| Pro | ₩9,900/월 | 전체 기능 |
| Enterprise | 커스텀 | 팀 + API |

### 비즈니스 규칙
- 무료 → 유료 업그레이드: 즉시 적용, 잔여일 비례 청구
- 다운그레이드: 현재 구독 만료 후 적용
- 해지: 기간 종료까지 서비스 유지 (즉시 만료 아님)
- 결제 실패 시: 3일/5일/7일 재시도 → 구독 정지

## 2. 기술 설계 — 백엔드

### DB 스키마
```sql
-- 구독 상태 관리
CREATE TABLE subscriptions (
  id            UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  user_id       UUID NOT NULL REFERENCES users(id),
  stripe_customer_id     VARCHAR NOT NULL UNIQUE,
  stripe_subscription_id VARCHAR UNIQUE,
  plan          VARCHAR NOT NULL DEFAULT 'free', -- free | pro | enterprise
  status        VARCHAR NOT NULL DEFAULT 'active', -- active | past_due | canceled | trialing
  current_period_start TIMESTAMPTZ,
  current_period_end   TIMESTAMPTZ,
  cancel_at_period_end BOOLEAN DEFAULT false,
  created_at    TIMESTAMPTZ DEFAULT now(),
  updated_at    TIMESTAMPTZ DEFAULT now()
);

-- 결제 이벤트 감사 로그 (불변)
CREATE TABLE payment_events (
  id              UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  user_id         UUID REFERENCES users(id),
  stripe_event_id VARCHAR NOT NULL UNIQUE, -- 중복 처리 방지
  event_type      VARCHAR NOT NULL,
  payload         JSONB NOT NULL,
  processed_at    TIMESTAMPTZ DEFAULT now()
);
```

### API 엔드포인트
```
POST /api/billing/checkout          → Stripe Checkout Session 생성
POST /api/billing/portal            → Stripe Customer Portal 세션 생성
POST /api/billing/webhook           → Stripe Webhook 수신 (서명 검증 필수)
GET  /api/billing/subscription      → 현재 구독 상태 조회
POST /api/billing/cancel            → 구독 해지 예약
POST /api/billing/reactivate        → 해지 취소
```

### Webhook 처리 (핵심)
```
수신 이벤트 → stripe.webhooks.constructEvent() 서명 검증
  → stripe_event_id로 중복 체크 (payment_events 테이블)
  → 이미 처리됨 → 200 OK 즉시 반환 (멱등성)
  → 신규 → 트랜잭션 내 처리

핵심 이벤트:
  customer.subscription.created   → 구독 활성화, 사용자 플랜 업데이트
  customer.subscription.updated   → 플랜 변경, 상태 동기화
  customer.subscription.deleted   → 구독 만료, 플랜 다운그레이드
  invoice.payment_succeeded        → 결제 성공 이메일 발송
  invoice.payment_failed           → 결제 실패 이메일 + 재시도 안내
  customer.subscription.trial_will_end → 트라이얼 종료 3일 전 알림
```

### 서비스 레이어 구조
```typescript
// SubscriptionService
createCustomer(userId)              // Stripe Customer 생성 + DB 저장
createCheckoutSession(userId, plan) // 결제 페이지 URL 생성
syncSubscriptionFromStripe(sub)     // Stripe 구독 → DB 동기화
handleWebhookEvent(event)           // 이벤트 라우팅 + 멱등 처리
cancelSubscription(userId)          // period_end에 해지 예약
```

## 3. 기술 설계 — 프론트엔드

### 컴포넌트 구조
```
/billing
  PricingPage           → 플랜 비교 + CTA
  CheckoutSuccessPage   → 결제 완료 (Stripe redirect)
  BillingDashboard      → 현재 구독 상태
    ├─ SubscriptionStatus   → 플랜명, 갱신일, 상태 배지
    ├─ InvoiceHistory       → 청구 내역 목록
    └─ PlanActions          → 업그레이드/다운그레이드/해지 버튼
```

### 결제 흐름 (Stripe Hosted Checkout 권장)
```
사용자가 "Pro 구독" 클릭
  → POST /api/billing/checkout → sessionId + url 수신
  → window.location.href = url (Stripe Checkout 페이지로 이동)
  → 결제 완료 → success_url 으로 redirect (/billing/success?session_id=...)
  → 성공 페이지에서 구독 상태 확인 후 대시보드 표시
```

> **왜 Stripe Elements 대신 Hosted Checkout?**  
> PCI DSS 범위를 Stripe 서버로 제한 → 우리 서버가 카드 데이터를 직접 다루지 않음.  
> 커스텀 UI가 필수라면 Stripe.js + Payment Element 사용 가능.

### 상태 관리
```typescript
// 구독 훅
const useSubscription = () => {
  // SWR/React Query로 GET /api/billing/subscription 폴링
  // 결제 완료 직후엔 webhook delay 고려해 낙관적 업데이트
}

// 게이팅 훅
const useFeatureGate = (feature: Feature) => {
  const { subscription } = useSubscription();
  return isFeatureAllowed(subscription.plan, feature);
}
```

### 에러 UX
- 결제 실패: 명확한 실패 사유 + "결제 정보 업데이트" 링크
- Customer Portal 연동: 카드 변경, 청구 주소, 인보이스 다운로드를 Stripe 호스팅 페이지에 위임

## 4. 보안 설계

### 결제 데이터 보호
| 항목 | 규칙 |
|------|------|
| 카드 번호 | **절대 서버에 저장 금지**. Stripe 토큰만 저장 |
| Webhook Secret | 환경변수로만 관리. 코드/로그에 노출 금지 |
| Stripe API Key | Secret key는 서버사이드 전용. Publishable key만 클라이언트 노출 |
| Customer ID | DB에만 저장, API 응답에서 불필요한 노출 최소화 |

### Webhook 보안 (필수)
```typescript
// 반드시 raw body 사용 (JSON.parse 전)
const sig = req.headers['stripe-signature'];
const event = stripe.webhooks.constructEvent(
  req.rawBody,           // ← Buffer/string, 파싱 전
  sig,
  process.env.STRIPE_WEBHOOK_SECRET
);
// 검증 실패 시 400 반환 — 처리하지 않음
```

### 접근 제어
- 모든 `/api/billing/*` 엔드포인트: 인증 미들웨어 필수
- 구독 조회: `userId === session.userId` 검증 (타인 구독 조회 불가)
- 관리자 기능(강제 플랜 변경): 별도 admin role 체크

### 감사 로그
- 모든 구독 변경 이벤트를 `payment_events` 테이블에 불변 기록
- 결제 실패, 해지, 플랜 변경은 별도 알림 이메일 발송

### Rate Limiting
- `/api/billing/checkout`: 사용자당 10req/min 제한 (중복 세션 생성 방지)
- `/api/billing/webhook`: IP 화이트리스트 + Stripe 서명으로 이중 검증

## 5. 구현 단계 (추천 순서)

```
Phase 1 — 기반 (1~2일)
  ✅ Stripe 계정 설정, Product/Price 생성
  ✅ DB 스키마 마이그레이션
  ✅ Stripe Customer 생성 + 저장 로직

Phase 2 — 결제 플로우 (2~3일)
  ✅ Checkout Session API
  ✅ Webhook 수신 + 멱등 처리
  ✅ 구독 상태 DB 동기화

Phase 3 — 프론트엔드 (2일)
  ✅ PricingPage + 업그레이드 버튼
  ✅ 결제 성공/실패 페이지
  ✅ BillingDashboard + Customer Portal 연동

Phase 4 — 안정화 (1~2일)
  ✅ 결제 실패 이메일 알림
  ✅ Feature gating (플랜별 기능 제한)
  ✅ Stripe Test Mode로 전체 플로우 E2E 테스트
  ✅ 프로덕션 전환 전 보안 체크리스트 점검
```

## 6. 테스트 체크리스트

```
□ 신규 구독 생성 → DB 상태 확인
□ 결제 성공 webhook → 플랜 활성화 확인
□ 결제 실패 webhook (카드 4000000000000341) → past_due 전환
□ 구독 해지 → period_end까지 서비스 유지
□ 해지 취소 → 정상 복구
□ 동일 webhook 이벤트 2회 전송 → 중복 처리 없음 확인
□ 인증 없이 /api/billing/* 접근 → 401 반환
□ 다른 사용자 구독 조회 시도 → 403 반환
```
