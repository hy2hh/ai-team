# 사용자 인증 시스템 설계

---

## 1. 전체 아키텍처

```
클라이언트 (React/Next.js)
    │
    ▼
API Gateway / BFF
    │
    ├── Auth Service          ← 인증 핵심 로직
    │     ├── JWT 발급/검증
    │     ├── Session 관리
    │     └── OAuth 연동
    │
    ├── User Service          ← 계정 데이터
    │     ├── 프로필
    │     └── 자격증명(Credential)
    │
    └── Notification Service  ← 이메일 발송
          ├── 비밀번호 재설정
          └── 이메일 인증
```

## 2. 데이터 모델

```sql
-- 계정 테이블
CREATE TABLE users (
  id            UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  email         VARCHAR(255) UNIQUE NOT NULL,
  email_verified BOOLEAN DEFAULT false,
  created_at    TIMESTAMPTZ DEFAULT now(),
  updated_at    TIMESTAMPTZ DEFAULT now(),
  deleted_at    TIMESTAMPTZ            -- Soft delete
);

-- 자격증명 (이메일/비밀번호 전용)
CREATE TABLE credentials (
  id              UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  user_id         UUID REFERENCES users(id) ON DELETE CASCADE,
  password_hash   VARCHAR(255) NOT NULL,  -- bcrypt, cost=12
  failed_attempts INT DEFAULT 0,
  locked_until    TIMESTAMPTZ,
  created_at      TIMESTAMPTZ DEFAULT now(),
  updated_at      TIMESTAMPTZ DEFAULT now()
);

-- OAuth 연동 계정
CREATE TABLE oauth_accounts (
  id             UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  user_id        UUID REFERENCES users(id) ON DELETE CASCADE,
  provider       VARCHAR(50) NOT NULL,   -- 'google' | 'github'
  provider_id    VARCHAR(255) NOT NULL,
  access_token   TEXT,                   -- 암호화 저장
  refresh_token  TEXT,                   -- 암호화 저장
  token_expires  TIMESTAMPTZ,
  created_at     TIMESTAMPTZ DEFAULT now(),
  UNIQUE(provider, provider_id)
);

-- 리프레시 토큰
CREATE TABLE refresh_tokens (
  id          UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  user_id     UUID REFERENCES users(id) ON DELETE CASCADE,
  token_hash  VARCHAR(255) UNIQUE NOT NULL,  -- SHA-256 해시
  user_agent  TEXT,
  ip_address  INET,
  expires_at  TIMESTAMPTZ NOT NULL,
  revoked_at  TIMESTAMPTZ,
  created_at  TIMESTAMPTZ DEFAULT now()
);

-- 비밀번호 재설정 토큰
CREATE TABLE password_reset_tokens (
  id          UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  user_id     UUID REFERENCES users(id) ON DELETE CASCADE,
  token_hash  VARCHAR(255) UNIQUE NOT NULL,
  expires_at  TIMESTAMPTZ NOT NULL,         -- 1시간 유효
  used_at     TIMESTAMPTZ,
  created_at  TIMESTAMPTZ DEFAULT now()
);

-- 이메일 인증 토큰
CREATE TABLE email_verification_tokens (
  id          UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  user_id     UUID REFERENCES users(id) ON DELETE CASCADE,
  token_hash  VARCHAR(255) UNIQUE NOT NULL,
  expires_at  TIMESTAMPTZ NOT NULL,         -- 24시간 유효
  used_at     TIMESTAMPTZ,
  created_at  TIMESTAMPTZ DEFAULT now()
);
```

## 3. API 설계

### 3-1. 이메일/비밀번호 인증

```
POST /auth/register
  Body: { email, password, name }
  → 계정 생성 + 이메일 인증 메일 발송
  → 201 Created

POST /auth/verify-email
  Body: { token }
  → 이메일 인증 완료
  → 200 OK

POST /auth/login
  Body: { email, password }
  → Access Token (15분) + Refresh Token (30일) 발급
  → 200 OK

POST /auth/logout
  Header: Authorization: Bearer {accessToken}
  → Refresh Token 폐기
  → 204 No Content

POST /auth/refresh
  Body: { refreshToken }  OR  Cookie: refresh_token
  → 새 Access Token 발급 (Rotation 방식)
  → 200 OK
```

### 3-2. OAuth

```
GET  /auth/oauth/{provider}
  → 해당 provider 인가 URL로 리다이렉트
  → state, PKCE code_challenge 생성 후 세션 저장

GET  /auth/oauth/{provider}/callback
  QueryParam: code, state
  → state 검증 → code 교환 → 사용자 정보 조회
  → 신규: 계정 생성 후 토큰 발급
  → 기존: 로그인 처리 후 토큰 발급
  → 프론트엔드로 리다이렉트 (토큰은 URL fragment 또는 short-lived code 사용)
```

### 3-3. 비밀번호 재설정

```
POST /auth/forgot-password
  Body: { email }
  → 이메일 존재 여부 무관하게 항상 200 반환 (열거 방지)
  → 재설정 링크 발송

POST /auth/reset-password
  Body: { token, newPassword }
  → 토큰 검증 → 비밀번호 변경 → 모든 세션 무효화
  → 200 OK
```

## 4. 토큰 전략

```
Access Token (JWT)
  - 유효기간: 15분
  - 저장: 메모리(React state) — XSS 방어
  - 갱신: Refresh Token으로 자동 갱신

Refresh Token
  - 유효기간: 30일
  - 저장: HttpOnly + Secure + SameSite=Strict 쿠키
  - 전략: Refresh Token Rotation
    → 사용할 때마다 새 토큰 발급 + 이전 토큰 폐기
    → 탈취 시 재사용 감지 → 해당 유저 전체 세션 폐기

JWT Payload 예시:
{
  "sub": "user_uuid",
  "email": "user@example.com",
  "role": "user",
  "iat": 1712600000,
  "exp": 1712600900,
  "jti": "unique-token-id"   ← 토큰 블랙리스트용
}
```

## 5. OAuth 플로우 상세

```
1. 프론트 → GET /auth/oauth/google
   서버: state = crypto.randomUUID()  (CSRF 방지)
         code_verifier = random 43-128자
         code_challenge = BASE64URL(SHA256(verifier))
         Redis에 state → { code_verifier, expiry } 저장
   서버 → 302 Google 인가 URL로 리다이렉트

2. Google → GET /auth/oauth/google/callback?code=xxx&state=yyy
   서버: state 검증 (Redis에서 확인)
         code + code_verifier → Google 토큰 교환
         Google UserInfo API 호출

3. 계정 매핑
   google_id 기존 존재? → 로그인
   email 기존 존재?     → 계정 연결 후 로그인
   신규?                → 계정 생성 후 로그인

4. 토큰 전달 (보안 방법)
   Option A: 서버가 short-lived code(30초) 발급
             → 프론트 /auth/exchange?code=xxx로 Access/Refresh 토큰 교환
   Option B: 서버 → 302 프론트 URL#access_token=xxx (fragment는 서버 미도달)
```

## 6. 비밀번호 재설정 플로우

```
1. POST /auth/forgot-password { email }
   → DB에서 user 조회
   → 토큰 생성: crypto.randomBytes(32) → URL-safe Base64
   → DB에 SHA-256(token) 저장 (원본은 저장 안 함)
   → 이메일 발송: https://example.com/reset?token=원본토큰
   → 응답: 항상 200 (이메일 존재 여부 노출 금지)

2. GET /reset?token=xxx (프론트)
   → 새 비밀번호 입력 UI 렌더

3. POST /auth/reset-password { token, newPassword }
   → DB에서 SHA-256(token) 검색
   → 만료 확인 (1시간)
   → 이미 사용됨 확인
   → bcrypt(newPassword, 12)로 해시 후 저장
   → used_at 업데이트
   → 해당 유저의 모든 refresh_tokens 폐기
   → 200 OK
```

## 7. 보안 요구사항

### 7-1. 비밀번호 정책

```
최소 길이: 12자
필수 포함: 대문자, 소문자, 숫자, 특수문자 중 3종 이상
금지 패턴:
  - 이메일 주소 포함
  - 연속 문자 (abc, 123, qwerty)
  - 유출된 비밀번호 목록 대조 (HaveIBeenPwned API 활용)

해싱: bcrypt, cost factor = 12
      (향후: argon2id 마이그레이션 권장)
```

### 7-2. Rate Limiting

```
로그인:          IP당 10회/10분 → 초과 시 15분 잠금
비밀번호 재설정: IP당 3회/시간
계정 잠금:       동일 계정 5회 실패 → 30분 잠금
OAuth 콜백:      IP당 20회/분

구현: Redis + Sliding Window Counter
헤더: X-RateLimit-Remaining, X-RateLimit-Reset
응답: 429 Too Many Requests + Retry-After 헤더
```

### 7-3. HTTPS / 전송 보안

```
TLS 1.2 이상 강제
HSTS: max-age=31536000; includeSubDomains; preload
쿠키: HttpOnly, Secure, SameSite=Strict
CORS: 허용 도메인 명시적 화이트리스트
```

### 7-4. 인젝션 방어

```
모든 DB 쿼리: Parameterized Query / ORM 사용
이메일 입력: 정규식 + 길이 제한(255자)
비밀번호: 최대 길이 제한(128자) — bcrypt DoS 방지
```

### 7-5. 감사 로그 (Audit Log)

```
기록 항목:
  - 로그인 성공/실패 (IP, User-Agent, 타임스탬프)
  - 비밀번호 변경
  - OAuth 계정 연결/해제
  - 비밀번호 재설정 요청
  - 강제 로그아웃 (전체 세션 폐기)
  - 계정 생성/삭제

보존 기간: 90일 이상
이상 감지: 새 국가/IP에서 로그인 시 이메일 알림
```

### 7-6. 추가 보안 레이어

```
이메일 인증 필수 (로그인 전 완료)
선택적 MFA (TOTP: Google Authenticator 호환)
세션 탈취 감지:
  - Refresh Token 재사용 감지 → 전체 세션 폐기
  - User-Agent 급격한 변경 감지
비밀번호 변경 시 → 모든 기존 세션 강제 로그아웃
계정 삭제: Soft delete + 30일 후 완전 삭제
```

## 8. 프론트엔드 보안

```
Access Token 저장:
  ✅ 메모리 (React 상태 또는 zustand)
  ❌ localStorage / sessionStorage (XSS에 노출)

자동 갱신 흐름:
  → 앱 로드 시 /auth/refresh 호출 (쿠키의 Refresh Token 사용)
  → Access Token 만료 1분 전 silent refresh
  → 401 응답 시 refresh 재시도 → 실패하면 로그인 페이지로

Silent Refresh 예시:
  interceptor에서 401 감지
    → refresh 요청 (대기 중인 요청은 queue에 보관)
    → 성공 시 queue의 요청 재실행
    → 실패 시 전체 queue 거부 + 로그아웃
```

  ✅ 이메일 인증

  ✅ Rate Limiting
  ✅ 감사 로그

  ✅ Google OAuth
  ✅ GitHub OAuth
  ✅ 계정 연결/해제
  ✅ 세션 목록 조회 + 개별 폐기

Phase 3
  ✅ TOTP MFA
  ✅ 이상 로그인 감지 알림
