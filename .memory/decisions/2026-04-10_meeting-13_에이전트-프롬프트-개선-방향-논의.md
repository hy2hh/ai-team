*[회의 #13] Krusty(Designer) — 에이전트 프롬프트 개선 의견*

*1. 핵심 의견*

현재 Krusty 프롬프트의 가장 큰 문제는 *시스템 프롬프트와 컨텍스트 파일 간 대규모 중복*입니다. 토큰 비용을 불필요하게 소모하고 있으며, 충돌 시 어느 쪽이 정본(SSOT)인지 혼란이 생깁니다. 또한 11단계 워크플로우가 모든 작업에 일괄 적용되는데, 간단한 토큰 수정과 전체 페이지 설계가 동일 프로세스를 타는 것은 비효율적입니다.

*2. 근거 (직접 확인한 사실)*

• *중복 데이터*: 시스템 프롬프트의 `절대 위반 금지` 섹션에 색상(`#0064FF`, `#F4F4F4`), 타이포(32px/24px/16px/13px), 카드 스펙(16px radius, 20px padding), 모션(Spring, 바텀시트, 카운트업), 여백 체계가 모두 나열됩니다. 이 값들은 `toss-design-system.md` §2~§6에 *동일하게* 정의되어 있습니다. 매 세션 시작 시 같은 정보를 시스템 프롬프트 + Read 파일로 *2회 로드*합니다.

• *SSOT 충돌 위험*: `conventions.md:17`에 "값 충돌 시 `component-guide.md` 기준 사용"이라고 명시되어 있지만, 시스템 프롬프트의 `절대 위반 금지`는 더 강한 어조("이 값 사용")로 되어 있어 *우선순위가 3개*(시스템 프롬프트 > component-guide > toss-design-system)로 모호해집니다.

• *워크플로우 오버헤드*: 11단계 중 Step 1(칸반), Step 3(토스 스펙 Read), Step 4(CSS 토큰 파일 생성)는 단순 색상 변경이나 회의 참석에도 강제됩니다. 실제로 지금 이 회의 응답에도 `toss-design-system.md`를 Read해야 하는 구조입니다.

• *Bart 핸드오프 후 피드백 루프 부재*: `conventions.md:59-71`에 핸드오프 체크리스트는 있지만, Bart가 구현 후 디자인 의도와 다를 때의 *리뷰 → 수정 요청 프로세스*가 정의되어 있지 않습니다.

*3. 리스크/우려사항*

• 시스템 프롬프트에서 토큰 값을 제거하면, 컨텍스트 파일 Read 실패 시(파일 경로 변경 등) *디자인 기준 없이 작업*하게 될 위험이 있습니다. 현재 중복이 일종의 fallback 역할을 하고 있는 셈입니다.
• 워크플로우를 작업 유형별로 분기하면 프롬프트 분기 로직이 복잡해질 수 있습니다.

*4. 추천 행동*

• *시스템 프롬프트 경량화*: `절대 위반 금지` 섹션에서 구체적 수치를 제거하고 "SSOT: `toss-design-system.md`" 참조 포인터만 남기되, *핵심 3개 값*(`#0064FF`, `#F4F4F4`, `#FFFFFF`)만 fallback으로 유지. 나머지 타이포·모션·여백은 컨텍스트 파일에 일원화.

• *워크플로우 2-track 분리*:
  — _Light track_ (토큰 수정, 단일 컴포넌트 변경, 회의 참석): Step 1(칸반) → 작업 → 완료
  — _Full track_ (새 페이지 설계, 디자인 시스템 변경): 현행 11단계 유지

• *디자인 리뷰 프로세스 추가*: Bart 구현 완료 후 Krusty가 스크린샷/코드 기반으로 디자인 QA하는 단계를 `conventions.md` §4 핸드오프 체크리스트 아래에 추가. "구현 → 디자인 리뷰 → 승인/수정 요청" 사이클 명시.

• *Closed Token Layer 검증 자동화*: 현재 `component-guide.md:58-80`의 자가 체크리스트가 수동 점검인데, Bart 구현물에 대해 Chalmers(QA)가 토큰 위반 여부를 자동 검증하는 테스트 케이스를 추가하면 디자인 일관성이 보장됩니다.
