• *프론트엔드 학습 도메인 분포*: React 패턴, CSS 함정, 번들 최적화, 접근성, 상태관리 — 대략 5개 도메인. 도메인당 2개면 10개로 충분

## 최종 결정
*회의 #10 종합 — Learnings 하드캡 정책 최종 결정*

---

*1. 합의된 사항 (전원 일치)*

• *필터링 품질이 캡 수치보다 중요* — 6명 전원이 이 점을 강조
• *컨텍스트 비용 영향은 미미* — 10개여도 기본 컨텍스트 대비 ~3-8% 증가, 60% 절감 목표와 충돌 없음
• *현재 데이터 0건* — 운영 데이터 없이 정밀한 차등화는 근거 없는 추측
• *confidence 기반 하한선 필수* — 저품질 학습 로드는 오히려 해로움

---

*2. 충돌 지점: 공통 vs 차등*

| 옵션 A (공통 10개) | 옵션 B (차등) |
| --- | --- |
| Bart, Homer, Lisa 지지 | Wiggum(7개), Chalmers(7개) 지지 |
| 장점: 단순, 관리 비용 0 | 장점: 역할별 최적화 |
| 단점: 과잉 로드 가능성 | 단점: 7개 에이전트 × 개별 설정 관리 |

*필터링 우선순위도 갈림:*
• Bart/Homer/Lisa: `relevance → confidence → recency`
• Wiggum: `confidence → recency → relevance` (보안은 확실성 최우선)

---

*3. 최종 결정*

*공통 10개 하드캡 (옵션 A) 채택 + Wiggum 예외 조건 수용*

근거:
• 3:2로 옵션 A 다수. 데이터 0건인 초기 단계에서 차등화는 과도한 최적화라는 Homer/Bart/Lisa 논거가 설득력 있음
• 다만 Wiggum의 `confidence 최우선` 논거는 타당 — 보안 학습은 불확실하면 안 쓰는 게 낫다. 이건 *캡 문제가 아니라 필터 문제*이므로 에이전트별 필터 순서 허용으로 해결

*확정 사양:*

```
하드캡: 전 에이전트 공통 10개
기본 필터 순서: relevance → confidence(≥7) → recency(30일)
예외: Wiggum은 confidence(≥8) → recency → relevance
Lisa source-registry: 하드캡과 별도 관리 (학습 10개 + 소스 5개)
Lisa type:source decay: 30일 → 60일 완화
재검토 시점: 전체 누적 50개 돌파 후 실데이터 기반
```

confidence 하한을 7로 설정 (Bart 6, Homer 5 제안보다 높음) — 초기에는 엄격하게 시작하고, 데이터 쌓이면 완화하는 것이 안전.
