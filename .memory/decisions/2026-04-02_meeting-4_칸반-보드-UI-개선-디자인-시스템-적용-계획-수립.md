### backend
Backend API 현황을 분석 완료했습니다.

---

*1. 핵심 의견*

현재 Backend API는 기본적인 CRUD와 드래그앤드롭을 지원하지만, *카드 상세 모달 및 고급 UI 기능에 필요한 일부 API가 누락*되어 있습니다. UI 개선 진행 전 API 보완이 필요합니다.

---

*2. 근거 (API 현황 분석)*

| 엔드포인트 | 상태 | 비고 |
| --- | --- | --- |
| `GET /boards/:id` | ✅ 준비됨 | 컬럼+카드 JOIN 쿼리 최적화 완료 (boards.ts:19-70) |
| `PATCH /cards/:id/move` | ✅ 준비됨 | 드래그앤드롭 + WIP limit 체크 (cards.ts:102-136) |
| `PATCH /cards/:id` | ✅ 준비됨 | 카드 수정 API (cards.ts:74-99) |
| `GET /cards/:id` | ❌ 미구현 | *단일 카드 상세 조회 없음* — 모달에서 필요 |
| Card Activity API | ❌ 미구현 | 타입만 정의됨 (types.ts:39-46), CRUD 없음 |
| Column CRUD | ❌ 미구현 | 컬럼 생성/수정/삭제 API 없음 |

*WebSocket 이벤트 (index.ts:35-42):*
- `card:created`, `card:updated`, `card:moved`, `card:deleted` 브로드캐스트 구현됨
- 실시간 동기화 기본 인프라는 준비 완료

---

*3. 리스크/우려사항*

• *모달 상세 조회 시 N+1 문제*: 현재는 `GET /boards/:id`로 전체 데이터를 가져오는 방식 — 카드 수가 많아지면 모달 열 때마다 과도한 데이터 전송
• *Activity 히스토리 미구현*: 카드 변경 이력 추적 불가 (types.ts에 `CardActivity` 정의만 있음)
• *Batch reorder 미지원*: 여러 카드 순서 동시 변경 시 N번 API 호출 필요

---

*4. 추천 행동*

UI 개선 작업 병행을 위해 아래 API 추가를 제안합니다:

```
우선순위 1 (모달 기능 필수)
├─ GET /cards/:id          — 단일 카드 상세 조회
└─ GET /cards/:id/activity — 카드 변경 히스토리

우선순위 2 (UX 개선)
├─ POST /cards/:id/activity — 댓글/활동 추가
└─ PATCH /cards/batch-move   — 다중 카드 순서 재배치

우선순위 3 (보드 관리)
└─ POST/PATCH/DELETE /columns — 컬럼 CRUD
```

*결론*: 현재 드래그앤드롭은 즉시 구현 가능하나, 카드 상세 모달 기능 구현 시 `GET /cards/:id` API가 필수입니다. Frontend 작업 착수 전 해당 API 추가를 먼저 진행하는 것을 권장합니다.

### 2. 충돌 분석

| 쟁점 | Designer 입장 | Frontend 입장 | Backend 입장 |
| --- | --- | --- | --- |
| *최우선 작업* | 타이포그래피 토큰 시스템화 | 모달 유틸리티 클래스 적용 (즉시 가능) | GET /cards/:id API 추가 |
| *Pretendard 폰트* | Phase 3 적용 권장 | 언급 없음 (현재 Geist Sans 유지 암묵적 동의) | — |
| *모달 상세 조회* | — | 현재 Board API로 충분 가정 | *단일 카드 API 필수* (N+1 문제) |

*핵심 충돌:* Frontend는 "즉시 실행 가능"이라 했으나, Backend는 "모달 기능 구현 시 GET /cards/:id 필수"라고 지적. 

*분석:*
- 현재 `GET /boards/:id`는 전체 보드 데이터를 반환 → 모달에서 이미 로드된 카드 데이터 재사용 가능
- 그러나 *Activity 히스토리*, *댓글 기능* 추가 시 단일 카드 API 필수
- *결론:* 기본 모달 스타일링은 API 없이 진행 가능, 고급 기능(Activity)은 API 선행 필요

### 3. 최종 결정

*접근 방식: 병렬 트랙 + 의존성 분리*

```
Track A (Designer + Frontend 협업)     Track B (Backend 단독)
─────────────────────────────────     ────────────────────────
Phase 1: 토큰 추가 정의               Phase 1: GET /cards/:id API
  - 타이포그래피 토큰                   - 단일 카드 상세 조회
  - Border Radius 표준화               
                                      
Phase 2: 모달 스타일 통일              Phase 2: Activity API (선택)
  - 유틸리티 클래스 적용                - GET/POST /cards/:id/activity
  - 접근성 강화                        
```

*결정 근거:*
1. *토큰 정의*와 *API 추가*는 독립적 → 병렬 진행으로 리드타임 단축
2. *모달 스타일링*은 기존 데이터로 즉시 가능 → Frontend 즉시 착수
3. *Pretendard 폰트*는 번들 사이즈 리스크로 *보류* (Geist Sans 유지)
4. *컬럼 내 카드 순서 변경*은 이번 스코프에서 *제외* (난이도 상, 별도 스프린트)
