## 🏗️ 백엔드 아키텍처 관점 의견

---

### 1. 핵심 의견

법률 서비스는 *데이터 민감도가 극도로 높은 도메인*입니다. 백엔드 관점에서 핵심은 (1) 변호사-의뢰인 비밀유지 수준의 *데이터 보안 아키텍처*, (2) LLM 응답의 *감사 추적(Audit Trail) 시스템*, (3) 대용량 법률 문서의 *벡터 검색 인프라* 구축입니다. 기능보다 신뢰성과 규정 준수가 선행되어야 합니다.

---

### 2. 핵심 기능 (백엔드 관점)

| 기능 | 설명 | 우선순위 |
| --- | --- | --- |
| *End-to-End 암호화 저장소* | 법률 문서/대화 내용 AES-256 암호화, 고객별 키 분리 | 필수 |
| *감사 로그 시스템* | 모든 AI 응답·문서 접근·수정 이력 불변 기록 (법적 분쟁 대비) | 필수 |
| *RAG 파이프라인* | 법률 DB(판례, 법령) 벡터화 + 실시간 검색 (Pinecone/Weaviate) | 필수 |
| *LLM 게이트웨이* | 멀티 프로바이더 관리, 비용 추적, Rate Limiting, Fallback | 필수 |
| *문서 버전 관리* | 계약서 리비전 추적, 변경 이력 비교 (Git-like) | MVP 후 |

---

### 3. 가장 큰 기술적/사업적 리스크

*기술적 리스크:*
• *LLM Hallucination* — 잘못된 법률 조언 시 손해배상 책임. 백엔드에서 출처 인용 강제 + Confidence Score 시스템 필요
• *데이터 유출* — 변호사-의뢰인 비밀유지 특권(Attorney-Client Privilege) 위반 시 서비스 종료 수준 리스크. 단일 DB 침해로 전체 고객 데이터 노출 방지 설계 필요
• *법령 DB 동기화* — 한국 법령정보센터 API 의존성. 법령 개정 반영 지연 시 잘못된 답변 생성

*사업적 리스크:*
• *변호사법 위반 가능성* — 비변호사의 법률사무 취급 금지(변호사법 제109조). API 응답에 "법률 자문 아님" 면책 조항 강제 삽입 필요
• *클라우드 규정* — 개인정보보호법, 정보통신망법 준수. 국내 리전(AWS 서울, NCP 등) 필수

---

### 4. MVP 범위 제안 (3개월)

```
Phase 1: MVP 아키텍처 (3개월)
├── 인프라
│   ├── AWS 서울 리전 (규정 준수)
│   ├── PostgreSQL + pgvector (벡터 검색)
│   └── Redis (세션/캐시)
├── 핵심 API
│   ├── /auth — JWT + MFA 인증
│   ├── /documents — 문서 업로드/암호화 저장
│   ├── /chat — LLM 대화 (감사 로그 포함)
│   └── /search — 법령/판례 RAG 검색
└── 보안
    ├── 전송 중 암호화 (TLS 1.3)
    ├── 저장 시 암호화 (AES-256)
    └── 감사 로그 (immutable append-only)
```

*MVP 제외 (Phase 2):*
• 멀티테넌시 로펌 기능
• 실시간 협업 편집
• 자동 계약서 생성
• 소송 일정 관리

### 5. 추천 행동

1. *즉시*: 법률 자문 범위 확인 — 변호사법 위반 리스크 법률 검토 선행 (법무법인 자문 필요)
2. *설계 단계*: 데이터 거주지(Data Residency) 정책 확정 — 국내 리전 필수 여부 결정
3. *MVP 전*: Penetration Test 및 보안 감사 일정 확보 — 런칭 전 필수
4. *아키텍처*: 감사 로그 불변성 보장 — 블록체인 또는 Append-Only DB 설계 검토

---

*요약:* 법률 AI는 기능보다 *신뢰*가 핵심입니다. "변호사 대체"를 목표로 하더라도 MVP에서는 "변호사 보조 도구"로 포지셔닝하고, 규정 준수와 보안 아키텍처를 선행 구축해야 합니다.

## 프론트엔드 관점 의견 — AI 법률 서비스

---

### 1. 핵심 의견

법률 서비스 UI의 핵심은 *복잡한 법률 문서를 직관적으로 탐색*하고, *AI와의 대화를 자연스럽게 이어가는 것*입니다. 사용자가 변호사든 일반인이든, 긴 계약서에서 핵심 조항을 빠르게 찾고 AI 분석 결과를 신뢰할 수 있어야 합니다.

---

### 2. 핵심 기능 (프론트엔드 관점)

| 기능 | 설명 | 기술적 고려사항 |
| --- | --- | --- |
| *대화형 AI 인터페이스* | 스트리밍 응답, 컨텍스트 유지, 출처 인용 표시 | SSE/WebSocket, 마크다운 렌더링 |
| *문서 뷰어/에디터* | 긴 문서 가상화, 조항별 하이라이팅, 주석 기능 | 가상 스크롤, PDF.js/문서 파싱 |
| *분석 결과 시각화* | 리스크 조항 강조, 비교 뷰, 요약 대시보드 | diff 뷰어, 차트 라이브러리 |
| *검색 인터페이스* | 법률 용어 자동완성, 판례/조문 필터링 | Debounce, 가상화된 드롭다운 |

---

### 3. 기술적 리스크

• *긴 문서 렌더링 성능*: 100페이지+ 계약서를 브라우저에서 부드럽게 렌더링하려면 가상화(virtualization) 필수. 무작정 DOM에 올리면 메모리 폭발
• *접근성 격차*: 법률 서비스는 시각장애인, 고령자도 사용. WCAG AA 준수 + 스크린 리더 호환 필수
• *클라이언트 보안*: 민감한 법률 문서가 브라우저 메모리에 남으면 안 됨. 세션 만료 시 메모리 정리, 복사/붙여넣기 제한 고려
• *AI 응답 신뢰도 표현*: "할루시네이션" 문제를 UI로 어떻게 완화할 것인가? (출처 인용, 신뢰도 배지, 면책 고지 등)

---

### 4. MVP 범위 (3개월 기준)

*포함 권장:*
• 핵심 채팅 UI (스트리밍 응답 + 출처 인용 표시)

*MVP 이후로 미루기:*
• 실시간 협업 편집
• 복잡한 문서 비교 뷰
• 오프라인 PWA 지원
• 다국어 지원

### 5. 추천 행동

1. *디자인 시스템 선정*: 법률 서비스 특성상 신뢰감 있는 UI 톤 필요 — 기존 디자인 시스템(예: Radix, shadcn/ui) 기반으로 커스터마이징 권장

## AI 법률 서비스 — 디자이너 관점 의견

---

### 1. 핵심 의견

법률 서비스의 UI/UX는 *신뢰(Trust)*와 *명확성(Clarity)*이 핵심입니다. 사용자는 법적 문제로 불안한 상태에서 서비스에 접근하므로, 복잡한 법률 정보를 쉽게 이해할 수 있는 *정보 계층 구조*와 AI 응답의 *신뢰도/한계를 명확히 표시*하는 UI가 반드시 필요합니다.

---

### 2. 핵심 기능 (디자인 관점)

| 기능 | UI/UX 요구사항 |
| --- | --- |
| *대화형 법률 상담 UI* | 법률 용어 툴팁, 출처 표시, 신뢰도 인디케이터 |
| *문서 업로드/분석 플로우* | 진행 상태 표시, 분석 결과의 시각적 하이라이트 |
| *계약서 작성 위자드* | 단계별 가이드, 입력 검증, 실시간 미리보기 |
| *면책 및 한계 고지 시스템* | 모든 AI 응답에 명확한 경고 표시 (법적 조언 아님 명시) |

---

### 3. 리스크/우려사항

• *신뢰 구축의 어려움*: AI 법률 조언에 대한 불신이 높음. "전문 변호사 검토" 배지나 출처 명시 등 신뢰 시그널 디자인이 필수
• *정보 과부하*: 법률 문서는 길고 복잡함 → 요약/하이라이트/단계적 공개(Progressive Disclosure) 패턴 필요
• *접근성*: 법률 서비스 사용자는 연령대가 넓음 (40-60대 포함). 최소 터치 타겟 48px, 폰트 16px 이상, 고대비 모드 필수
• *오류 시 책임 문제*: AI 실수 시 책임 소재 → 모든 화면에 면책 고지 UI 배치 필요

---

### 4. MVP 범위 추천 (3개월)

*단일 핵심 플로우에 집중*하는 것을 권장합니다:

```
법률 Q&A 챗봇 UI (1개 Task Flow)
├─ 질문 입력 + 카테고리 선택
├─ AI 응답 + 출처/신뢰도 표시
├─ 관련 법조문 링크
└─ 면책 고지 배너 (상시 노출)
```

• *포함*: 대화 UI, 법률 용어 사전 팝오버, 기본 다크/라이트 모드
• *제외*: 문서 업로드, 계약서 작성 위자드 (v2로 이관)
• *디자인 시스템*: 차분하고 신뢰감 있는 톤 — Navy/Dark Blue 계열 Primary, 과도한 장식 배제

---

### 5. 추천 행동

1. *사용자 리서치 우선*: 실제 법률 상담 니즈가 있는 사용자 5-10명 인터뷰 → 핵심 Pain Point 파악
2. *경쟁사 UI 분석*: LawTalk, 로톡, 미국 DoNotPay 등의 UX 패턴 벤치마킹
3. *Low-fi 프로토타입*: 챗봇 UI 와이어프레임 → 사용성 테스트 후 High-fi 진행
4. *디자인 토큰 정의*: 법률 서비스에 맞는 컬러·타이포 시스템 선정 (신뢰감 + 가독성)

## 🔒 SecOps 의견 — AI 법률 서비스 보안 아키텍처

### 1. 핵심 의견

법률 데이터는 *변호사-의뢰인 비밀유지 특권*이 적용되는 최고 수준의 민감 정보입니다. AI 법률 서비스는 *보안이 기능보다 선행*해야 하며, 단 한 건의 데이터 유출도 사업 존폐를 결정할 수 있습니다. MVP 단계부터 보안을 핵심 기능으로 설계해야 합니다.

---

### 2. 핵심 보안 기능 (반드시 포함)

| 기능 | 필요성 | 구현 수준 |
| --- | --- | --- |
| *강력한 인증* | 무단 접근 차단 | MFA 필수, 세션 관리 |
| *End-to-End 암호화* | 전송/저장 데이터 보호 | TLS 1.3 + AES-256 |
| *RBAC/ABAC 접근 제어* | 최소 권한 원칙 | 문서별 접근 권한 |
| *상세 감사 로그* | 법적 증거, 컴플라이언스 | 누가/언제/무엇을 열람했는지 |
| *테넌트 격리* | 의뢰인 간 데이터 분리 | DB/스토리지 레벨 격리 |
| *LLM 보안 레이어* | 프롬프트 인젝션 방어 | 입력 검증, 출력 필터링 |

---

### 3. 가장 큰 리스크

*기술적 리스크:*
• *프롬프트 인젝션 공격* — 악의적 입력으로 다른 의뢰인의 법률 정보 유출 가능
• *LLM Hallucination* — 잘못된 법률 자문 제공 시 손해배상 책임 발생
• *데이터 유출* — 법률 문서 노출 시 의뢰인 소송, 변호사 징계, 사업 종료

*사업적/법적 리스크:*
• *변호사법 위반* — 한국에서 무자격 법률 자문은 형사처벌 대상 (변호사법 제109조)
• *개인정보보호법 준수* — 민감정보 처리 동의, 보유기간, 파기 의무
• *전자서명법/전자문서법* — 법적 효력 있는 문서 처리 시 요건 충족 필요

---

### 4. MVP 보안 범위 (3개월)

```
[필수 — Day 1]
├── MFA 인증 (TOTP 또는 SMS)
├── TLS 1.3 전송 암호화
├── AES-256 저장 암호화
├── 기본 RBAC (관리자/사용자/게스트)
└── 감사 로그 (로그인, 문서 접근, AI 질의)

[권장 — MVP 내]
├── 입력 검증 + 프롬프트 인젝션 필터
├── 세션 타임아웃 (15분 비활성)
├── IP 기반 이상 접근 탐지
└── 데이터 백업 + 암호화된 복구

[Phase 2로 연기 가능]
├── ABAC (문서 레벨 세분화)
├── DLP (Data Loss Prevention)
└── SOC 2 / ISMS 인증 준비
```

---

### 5. 추천 행동

1. *법률 자문 면책 고지 필수* — "이 서비스는 법률 정보 제공이며, 변호사의 법률 자문을 대체하지 않습니다" 명시
2. *변호사 파트너십* — 무자격 법률 자문 리스크 회피를 위해 변호사 감수/협업 구조 설계
3. *보안 아키텍처 리뷰* — MVP 설계 단계에서 위협 모델링 (STRIDE) 수행
4. *침투 테스트* — 출시 전 LLM 보안 포함 모의 해킹 필수
5. *개인정보 영향평가* — 민감정보 처리 시 법적 요건 (PIPA 제33조)

---

*요약:* AI 법률 서비스는 "보안 실패 = 사업 실패"입니다. MVP에서도 인증/암호화/감사로그는 협상 불가 기능이며, 변호사법 리스크 회피를 위한 법적 구조 설계가 기술만큼 중요합니다.

### 2. 관점 차이 분석 (충돌 아닌 보완 관계)

| 에이전트 | MVP 핵심 제안 | 관점 |
| --- | --- | --- |
| Researcher | 판례 검색 + 계약서 검토 | 시장 검증된 수요 기반 |
| Backend | RAG 파이프라인 + 감사 로그 | 인프라/신뢰성 중심 |
| Frontend | 채팅 UI + PDF 뷰어 | 사용자 경험 중심 |
| Designer | Q&A 챗봇 단일 플로우 집중 | 범위 최소화 + 신뢰 UI |
| SecOps | 인증/암호화/접근제어 | 보안 아키텍처 중심 |

*분석:* 충돌이 아니라 *레이어별 우선순위 제안*입니다. 종합하면 하나의 일관된 MVP가 도출됩니다.

---

### 3. 최종 결정 — MVP 범위 (3개월)

```
🎯 핵심 가치 제안: "변호사의 240시간 절감 파트너"

[P0 — 필수 기능]
├── 대화형 AI 상담 UI
│   ├── 스트리밍 응답 + 출처 인용 표시
│   ├── 법률 용어 툴팁
│   └── 면책 고지 배너 (상시 노출)
├── 판례/법령 RAG 검색
│   ├── PostgreSQL + pgvector
│   └── 법령정보센터 API 연동
└── 보안 기반
    ├── MFA 인증
    ├── TLS 1.3 + AES-256 암호화
    └── 감사 로그 (불변)

[P1 — MVP 내 권장]
├── 문서 업로드 + PDF 뷰어 (하이라이팅 1종)
├── 계약서 리스크 조항 하이라이트
└── 기본 접근성 (키보드 탐색, 16px+ 폰트)

[Phase 2 이후]
├── 실시간 협업 편집
├── 계약서 자동 생성
├── 일반인 대상 법률 Q&A (규제 완화 후)
└── SOC 2 / ISMS 인증
```

*결정 근거:*
• 슈퍼로이어가 "AI 법률 비서" 포지셔닝으로 5개월 만에 변호사 14% 확보 — 검증된 B2B 수요
• 변호사법 리스크 회피를 위해 "정보 제공 + 변호사 검토 필수" 구조 채택
• 보안 사고 1건 = 사업 종료이므로 보안은 협상 불가
