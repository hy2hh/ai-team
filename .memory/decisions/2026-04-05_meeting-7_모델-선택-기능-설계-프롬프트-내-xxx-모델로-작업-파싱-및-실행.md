## Backend 의견 — 모델 선택 기능 설계

### 1. 핵심 의견

*구조화된 모델 선택(API 파라미터)을 기본으로 하고, 프롬프트 파싱은 보조 기능으로 제공하는 하이브리드 방식을 권장합니다.* 프롬프트 파싱만 의존하면 모델명 변형("gpt4o", "GPT-4o", "지피티포오"), 오타, 신규 모델 추가 시 파싱 로직 유지보수 비용이 급증합니다. 구조화된 선택은 API 계약이 명확하여 안정성과 테스트 용이성이 높습니다.

### 2. 근거

*안정성 비교:*
| 방식 | 장점 | 단점 |
| --- | --- | --- |
| 구조화된 선택 | 명확한 API 계약, 타입 안전, 테스트 용이 | 사용자 UX 추가 클릭 필요 |
| 프롬프트 파싱 | 자연스러운 사용자 경험 | 정규식/NLP 유지보수, 오인식 위험 |

*지원 모델 범위 권장:*
• *OpenAI*: gpt-4o, gpt-4-turbo, gpt-3.5-turbo
• *Anthropic*: claude-opus-4, claude-sonnet-4, claude-haiku
• *Google*: gemini-1.5-pro, gemini-1.5-flash

*모델 라우팅 아키텍처:*
```
┌─────────────┐     ┌──────────────┐     ┌─────────────────┐
│  API Layer  │────▶│ Model Router │────▶│ Provider Adapter│
│ (선택 수신) │     │ (검증/분기)  │     │ (OpenAI/Claude) │
└─────────────┘     └──────────────┘     └─────────────────┘
```

• *Model Router*: 요청의 `model` 필드 검증, 권한 확인, 해당 Provider로 분기
• *Provider Adapter*: 각 API의 인터페이스 차이를 추상화하는 어댑터 패턴
• *Config 기반 확장*: 신규 모델 추가 시 코드 수정 없이 config 추가만으로 가능하게 설계

### 3. 리스크/우려사항

• *비용 관리*: 모델별 가격 차이가 큼 (opus vs haiku 약 15배). 사용량 제한/과금 정책 필요
• *Rate Limit 분산*: 각 Provider별 rate limit이 다르므로 per-provider circuit breaker 필요
• *Fallback 전략*: 선택 모델 장애 시 대체 모델로 자동 전환할지 에러 반환할지 정책 결정 필요
• *프롬프트 파싱 오인식*: "클로드 무니에 레스토랑 검색해줘"를 Claude 모델로 잘못 해석할 위험

### 4. 추천 행동

1. *Phase 1*: 구조화된 API 파라미터(`model: "gpt-4o"`) 방식 먼저 구현
2. *Phase 2*: 프롬프트 파싱은 명시적 트리거 패턴만 지원 (예: `@gpt-4o`, `[claude-opus]`)
3. *Config 스키마*: 지원 모델 목록을 DB/config에서 관리하여 런타임 확장 가능하게
4. *비용 제어*: 모델별 일일 사용량 한도 및 사용자별 권한 레벨 설계
### frontend
## 프론트엔드 의견 — 모델 선택 UI 설계

### 1. 핵심 의견

*명시적 드롭다운 UI를 기본으로 제공하고, 자연어 파싱은 보조 기능으로 병행해야 합니다.* 자연어 파싱만으로는 사용자가 어떤 모델이 사용 가능한지 알 수 없어 UX가 불안정합니다. "발견 가능성(discoverability)"이 핵심입니다.

### 2. 근거

| 기준 | 드롭다운 UI | 자연어 파싱만 |
| --- | --- | --- |
| 발견 가능성 | 사용 가능한 모델 목록 즉시 확인 | 사용자가 모델명을 이미 알아야 함 |
| 오류 방지 | 선택지 제한으로 오타 불가 | "gpt4o", "GPT-4O", "gpt 4 o" 등 변형 처리 복잡 |
| 상태 피드백 | 현재 선택된 모델 시각적 확인 | 파싱 성공 여부 불투명 |
| 접근성 | WCAG 준수 키보드/스크린리더 지원 용이 | 자연어 의존은 보조기기 사용자에게 불리 |
| 학습 곡선 | 신규 사용자도 즉시 사용 가능 | 파워 유저 전용 |

### 3. 리스크/우려사항

• *자연어 파싱 단독 의존 시*: 모델명 동의어/오타 처리 로직이 복잡해지고, 파싱 실패 시 사용자 혼란 유발
• *드롭다운만 제공 시*: 빠른 입력을 원하는 파워 유저에게 불편 — 타이핑 흐름 중단
• *두 방식 충돌 가능성*: 드롭다운에서 GPT-4o 선택 후 프롬프트에 "Claude로 분석해줘" 입력 시 우선순위 혼란

### 4. 추천 행동

```
┌─────────────────────────────────────────────────┐
│ 모델: [GPT-4o ▼]  ← 드롭다운 (기본값 설정 가능) │
├─────────────────────────────────────────────────┤
│                                                 │
│  프롬프트 입력 영역                              │
│  "이 계약서를 검토해줘"                          │
│                                                 │
└─────────────────────────────────────────────────┘
```

• *프롬프트 입력창 상단*에 모델 선택 드롭다운 배치 — 입력 전 선택 유도
• 자연어에서 모델명 감지 시 *토스트 알림*으로 확인: "Claude Opus로 전환할까요? [확인] [취소]"
• 사용자별 *기본 모델 저장* 기능 제공 (localStorage 또는 사용자 설정)
• 드롭다운에 모델별 *간단한 특성 힌트* 표시: "GPT-4o — 빠름", "Claude Opus — 정교함"

## 회의 #7 종합 — 모델 선택 기능 설계 결정

### 1. 합의된 부분

*전원 합의:* *"드롭다운 UI 기본 + 자연어 파싱 보조"* 하이브리드 방식

| 합의 항목 | 근거 |
| --- | --- |
| 드롭다운 UI가 기본 | 발견 가능성, 오타 방지, 접근성 확보 — 업계 표준 (ChatGPT/Claude/Gemini 모두 채택) |
| 자연어 파싱 단독 의존 금지 | 모델명 변형/오타 처리 복잡, 오인식 리스크 ("클로드 무니에" → Claude 오인식) |
| Phase별 점진적 구현 | MVP → 보조 기능 → 스마트 라우팅 순서로 안정성 확보 |
| Config 기반 모델 목록 관리 | 신규 모델 추가 시 코드 수정 없이 확장 가능 |

### 2. 세부 구현 방식 — 입장 분석

| 쟁점 | Backend 제안 | Frontend 제안 | 분석 |
| --- | --- | --- | --- |
| 자연어 트리거 방식 | 명시적 패턴만 지원 (`@gpt-4o`, `[claude-opus]`) | 자연어 감지 시 토스트로 확인 요청 | *상충 아님* — 조합 가능 |
| 충돌 시 우선순위 | API 파라미터 우선 | UI 선택 후 프롬프트 충돌 시 확인 필요 | Frontend 안이 UX에 적합 |

*결론:* 두 방식 모두 채택. 명시적 패턴(`@model`)은 즉시 전환, 암묵적 언급("Claude로 해줘")은 토스트로 확인.

| *Phase 3* | 작업 유형별 스마트 라우팅 (선택 사항) | 추후 결정 |
